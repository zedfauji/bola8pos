<objective>
Wire the read/write data layer for tip distribution: extend the settings entity so the config round-trips through the existing useSettings/useMutationUpdateSetting infrastructure (with a 34/33/33 default), and add a useTipDistributionEntry(cajaSessionId) read hook to the caja entity that returns the immutable entry row for the report panel.

Purpose: This keeps the Settings tab and report panel thin — they consume entity hooks instead of touching Supabase directly (FSD). The settings extension reuses the generic settings-table pattern (19-RESEARCH.md Pattern 1), so admin-only write is already enforced by existing RLS. The caja hook mirrors the existing useCajaEntries read pattern.

Output: extended settings entity (queries/types + barrels) + one new caja read hook (+ barrels). Unit-verifiable via typecheck/lint; live reads depend on Plan 03's push (same wave) and are exercised end-to-end in Plan 06.
</objective>
