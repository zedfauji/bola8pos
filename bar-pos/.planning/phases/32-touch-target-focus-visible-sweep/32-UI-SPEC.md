# Phase 32 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Phase type: rollout, not redesign.** This phase changes sizing (`touchSize`), spacing (`gap-*`), and a single new focus-ring prop (`focusEmphasis`) on **existing** components across 6 pages. No new components, no new colors, no new copy, no control reordering. Every table below either points at pre-existing project tokens (verify-only) or gives the exact new class contract this phase introduces.

## Spacing Scale

Declared values (must be multiples of 4) — unchanged app-wide scale, referenced for the grid-gap audit (D-06):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding |
| sm | 8px | **Minimum enforced gap between adjacent tappable cards/buttons (D-06 floor)** |
| md | 16px | Default element spacing — current `PoolTableGrid` gap (`gap-4`, already compliant) |
| lg | 24px | Section padding |
| xl | 32px | Layout gaps |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level spacing |

**Grid-gap audit contract (TOUCH-03 / D-05 / D-06 / D-07):**
- No stepper components exist in scope — do not build stepper spacing rules. D-05 confirms this.
- Every tappable-card/button grid on the 6 in-scope pages must use `gap-2` (8px) or greater. Verify, do not assume:
  - `src/widgets/PoolTableGrid/index.tsx:159,173` — currently `gap-4` (16px) — **verify only, no change expected**, flag as regression if found otherwise.
  - KDS order-card grid (`src/pages/kds/`) and kds-bar order-card grid (`src/pages/kds-bar/`) — inspect actual markup for gap class; if `<gap-2` or unset, bump to `gap-2` minimum.
  - Kitchen-prep batch grid (`src/pages/kitchen-prep/`, `PrepBatchPreview.tsx`) — same audit.
  - Per-card action buttons inside KDS/kds-bar cards (Start/Ready/Bump) — verify actual stacking/full-width layout in code (D-07); if buttons sit side-by-side, they must also clear the 8px gap floor.
- Where a bump is needed, use Tailwind `gap-2` (exact class), not an arbitrary value.

Exceptions: none.

## Color

No new colors introduced. This phase reuses the existing `--ring` design token at full opacity for the emphasized focus ring (D-09) — do not introduce a new color value.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | existing app background tokens (unchanged) | Background, surfaces |
| Secondary (30%) | existing app card/nav tokens (unchanged) | Cards, sidebar, nav |
| Accent (10%) | existing `--ring` token: `oklch(0.708 0 0)` light / `oklch(0.556 0 0)` dark (`src/app/globals.css:28,70`) | **Focus rings only** — baseline (`ring-3 ring-ring/50`, D-08, verify-only) and emphasized (`ring-4 ring-ring` full opacity, D-09, new) |
| Destructive | existing `--destructive` token (unchanged, used by `variant="destructive"`) | Destructive actions only (void/cancel/delete flows already use `variant="destructive"` where applicable — not modified by this phase beyond touch-sizing/focus-ring) |

Accent reserved for: focus-visible ring outlines only — baseline ring-3/50% opacity on every focusable control app-wide (pre-existing, D-08), emphasized ring-4/100% opacity on exactly 3 controls (D-09/D-10, see Focus Ring Contract below). No other UI element may consume the accent token as part of this phase.

## Component Sizing & Focus Contract (this phase's actual deliverable)

### Touch-target sizing (D-01 – D-04)

| Component | Rule |
|-----------|------|
| Every raw shadcn `Button` / icon-button on the 6 in-scope pages | Convert to `POSButton` with `touchSize="default"` (44px, `min-h-[44px]`) at minimum — no exceptions, including icon-only buttons and dense-table row actions (`InventoryPagePanel` DataTable row actions) |
| `PoolTableGrid` line 120 raw `Button` | Convert to `POSButton touchSize="default"` (or higher per the table below) — D-02 |
| Destructive/irreversible actions: void/cancel pool session, stop-and-move table (2 targets — "delete inventory batch" dropped, not a shipped feature; rendered via `ConfirmDialog`/`AlertDialogAction`, not `POSButton` — needs passthrough props, see RESEARCH.md Pattern 3) | `touchSize="xl"` (72px, `min-h-[72px] text-lg font-semibold`) |
| Frequent/primary actions: add stock, notify waitlist, seat party, start timer, save prep batch, bump KDS order | `touchSize="large"` (56px, `min-h-[56px] text-base`) |
| Everything else (not critical, not frequent/primary) | `touchSize="default"` (44px floor) — no smaller |

No fixed-layout exceptions exist (D-04) — every control in scope can grow to its assigned size without a layout constraint override.

### Focus ring escalation (D-08 – D-10)

| State | Classes | Scope |
|-------|---------|-------|
| Baseline focus-visible (FOCUS-01) | `focus-visible:border-ring focus-visible:ring-3 focus-visible:ring-ring/50` (existing base CVA in `src/shared/ui/button.tsx:8`) | App-wide, already satisfied — **verify only**: confirm it survives the `touchSize` className merge (POSButton's `cn()` must not drop these classes) |
| Emphasized focus-visible (FOCUS-02, new) | New `focusEmphasis="high"` variant on the **base** `buttonVariants` CVA in `src/shared/ui/button.tsx` swaps to `focus-visible:ring-4 focus-visible:ring-ring` (full-opacity token, +1px ring width vs. baseline) | Applied to exactly 2 controls, the same set as the `xl` touch-target set: void/cancel pool session, stop-and-move table. No broader application. |

`focusEmphasis` is a prop on the shared `Button` (not `POSButton`-only) so later phases can reuse it — Phase 32 only wires it onto the 2 controls above.

### Tab order verification (FOCUS-03, D-11 – D-13)

| Surface | Verification |
|---------|--------------|
| `ManagerPinDialog` PIN entry flow as reached from `TableStatusPanel` | Automated Playwright assertion of Tab-key focus order matching visual order |
| Inventory `DataTable` search/filter row | Same |
| Other inventory/kitchen-prep form inputs (batch qty, adjustment-reason fields) | Same |

Method: new spec `e2e/44-focus-tab-order.spec.ts` (D-13, renumbered from 43 — that number is taken by `e2e/43-promotions.spec.ts`) — not manual click-through. Add to the 22-spec list in `CLAUDE.md` §E2E Test Suite once created (becomes the 23rd permanent spec).

Not in scope: a blanket sweep of every input on every page beyond the three named surfaces (D-12).
