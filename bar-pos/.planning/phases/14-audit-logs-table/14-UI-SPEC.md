# Phase 14 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Phase type: completion, not greenfield.** Most visual primitives already exist and are verified good in the codebase (`JsonDiffViewer`, `Sheet` pattern via `RefundSheet`/`SplitTabSheet`, `DataTable`, `PINKeypad`, `DateRangePicker`). This contract's job is almost entirely (a) composition rules for assembling those primitives into `/audit`, (b) the exact DOM/a11y contract already locked by `e2e/38-audit-logs.spec.ts` (treat as executable spec, not a suggestion), and (c) two genuinely new surfaces that have zero prior art: the Staff-page "Force PIN Change" trigger and the staff-side forced-PIN-change screen.

| Property | Value |
|----------|-------|
| Tool | shadcn (already initialized — `components.json` present) |

No new shadcn components need to be installed for this phase — every primitive required (`Sheet`, `Select`, `Table`, `AlertDialog`, `Button`, `Badge`, `Checkbox`, `Input`) already exists in `src/shared/ui/`.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px (`text-sm`) | Regular (400) | 1.5 |
| Label | 12px (`text-xs`) | Semibold (600) — uppercase tracking-wide for column headers, matches `JsonDiffViewer`'s "Before"/"After" labels | 1.4 |

Do not introduce `font-medium` (500) or `font-bold` (700) in new Phase 14 components even though older parts of the codebase use them — this phase's new surfaces standardize on the 2-weight contract.

- Page heading "Audit Log" (matches E2E `getByRole('heading', { name: 'Audit Log' })`): Display, 24px semibold.
- Sheet title (action name, e.g. "payment.process"): Heading, 20px semibold, monospace NOT required (plain sans, matches existing `SheetTitle` usage in `RefundSheet`).
- Table cell text (action, entity_type, actor name, timestamp): Body, 14px regular; action column uses `font-mono text-xs` ONLY inside the diff tree (JsonDiffViewer already does this) — the outer table row itself uses normal sans body text, not monospace.
- Forced-PIN-change screen staff name/heading: reuse `PINLoginForm`'s exact existing pattern — `text-xl font-bold` avatar-initial circle + `<h2 className="text-xl font-bold">`. This is pre-existing code (not new), so it is exempt from the 2-weight rule above — do not refactor it as part of this phase.

## Screen-by-Screen Contract

### A. `/audit` page (`pages/audit/index.tsx`)

Thin FSD container per convention (see `pages/reports/index.tsx` for the shape): renders `<SectionHeader>` (heading = "Audit Log", 24px semibold, matches E2E) + `<AuditLogTable />` widget. No logic in the page file itself.

Layout: `mx-auto max-w-6xl space-y-8` container, matching `StaffDashboard`'s exact wrapper classes for visual consistency with sibling manager+ pages.

### B. `AuditLogFilterBar` (inline or separate file inside `widgets/AuditLogTable/`)

Filter bar sits directly under the page heading, `flex flex-wrap items-center gap-2` (matches `DateRangePicker`'s internal layout), rendered via `DataTable`'s existing `toolbar` prop — do not build a second wrapper div, pass this component straight into `<DataTable toolbar={<AuditLogFilterBar .../>} .../>`.

Five filter controls, left to right:
1. **Action** — `<Select>` populated from `AuditActionSchema.options` (20 values from `audit-actions.ts`), trigger MUST have `id="audit-filter-action"` (locked by E2E, exact id, do not vary). Placeholder: "All actions".
2. **Entity type** — `<Select>`, options derived from distinct entity-type prefixes already implied by `AuditActionSchema` (`payment`, `tab`, `caja_session`, `order`, `combo`, `inventory`, `prep_production`, `permission`, `staff`, `settings`) — hardcode this list in the widget, no new DB query needed. Placeholder: "All entities".
3. **Actor** — `<Select>` populated from `useStaffList()` (already used in `ManagerPinDialog`/`StaffDashboard`), display staff name, value = staff id. Placeholder: "All staff".
4. **Date range** — two native `<input type="date">` fields styled exactly like `DateRangePicker.tsx`'s inputs (`rounded-md border border-input bg-background px-2 py-1 text-sm focus-visible:ring-2 focus-visible:ring-ring`), but with explicit ids: `id="audit-filter-date-from"` and `id="audit-filter-date-to"` (the first is locked by E2E, exact id, do not vary; the second follows the same naming convention for consistency). Do NOT reuse `DateRangePicker` unmodified since it has no `id` prop today — either add an `id` prop to `DateRangePicker` or build the two inputs inline in `AuditLogFilterBar`; either is acceptable, but the rendered DOM ids are non-negotiable.
5. **Free text search** — reuse `SearchInput` (already used via `DataTable`'s `searchable` prop elsewhere) OR a plain labeled `<Input>` — searches `entity_id`/`action` per the existing `useAuditLogs` query. Placeholder: "Search by entity ID or action…".

"Apply filters" button (`Button` size default, accent/primary variant) sits at the end of the filter row — filters are staged in local state and only committed to the `useAuditLogs(filters)` call when this button is clicked (not on every keystroke/select-change), matching the E2E's explicit "click Apply filters" step in every filter test.

**Security note (carried from RESEARCH.md V5 finding):** before wiring the free-text search filter to `useAuditLogs`, escape or strip `,` and `.` characters from the user-entered string — these are PostgREST `.or()` filter-string metacharacters and the current `entities/audit-log/model/queries.ts` implementation interpolates the raw string unescaped. This is a functional/security fix, but it belongs in this UI-SPEC because the input's `maxLength`/inline validation message ("Some characters aren't searchable and will be ignored") is a copywriting/UX decision, not just a backend one — flag to planner/executor.

### C. `AuditLogTable` — row + cell DOM contract (locked by E2E, treat as executable spec)

Built on top of `DataTable`, NOT a from-scratch table. Required DOM contract per row:
- A `<TableCell>` containing the raw action string (e.g. `payment.process`) — MUST be reachable via `page.getByRole('cell', { name: 'payment.process' })`, i.e. render the action label as plain text content of a `<td>`, not inside a nested interactive element that would change its accessible name.
- The same row MUST also be reachable via `page.getByRole('button', { name: /view diff for .* on .*/i })` — i.e. an accessible name of the exact shape `"View diff for {action} on {formattedDate}"`.

**Known DataTable gap (flag to planner):** `DataTable.tsx`'s `TableRow` currently has no per-row `aria-label`/`role` override hook — only `onRowClick`, `getRowClassName`. To satisfy both requirements above without breaking table semantics (overriding `role="button"` on a `<tr>` can strip its `role="row"`/children's `role="cell"` from the accessibility tree in some browsers), do ONE of:
  - (a) Extend `DataTable` with an optional `getRowAriaLabel?: (row: T) => string` prop that sets `aria-label` on the `<tr>` WITHOUT overriding its role (native `<tr>`/`<td>` keep implicit table roles even with a custom `aria-label`), then pass `getRowAriaLabel={(row) => \`View diff for ${row.action} on ${formatAuditDate(row.createdAt)}\`}` from `AuditLogTable`. **This is the recommended approach** — additive change to a shared component, keeps `role="cell"` intact, and a `<tr>` with `aria-label` is exposed as `role="row"` with an accessible name matching neither "row" role by default in Playwright's `getByRole('button', ...)` query... **verify at implementation time**: Playwright's role queries key off the *computed ARIA role*, so a plain `<tr>` (role="row") will NOT match `getByRole('button', ...)` regardless of `aria-label`. Given this, the row itself cannot literally be both `role="cell"`'s parent AND satisfy a `button` role query without an explicit `role="button"` somewhere in the row.
  - (b) **Corrected recommendation:** add a visually-hidden (`sr-only`) `<button>` element inside one cell (e.g., the first cell, layered behind/alongside the visible action text) whose accessible name is `"View diff for {action} on {date}"` and whose `onClick` triggers the same `onRowClick` handler — this satisfies the E2E's button-role query literally while the visible cell text satisfies the `cell` query, and avoids any risk of breaking table accessibility semantics. Alternatively, make the ENTIRE row clickable via `onRowClick` (existing `DataTable` behavior) for mouse/touch users, and layer the `sr-only` button only for the accessible-name contract — do not make the sr-only button the sole click target, since real users click the visible row.
- Timestamp format for the aria-label's date portion: reuse the same format as `PINLoginForm`'s clock displays where reasonable, or a full date+time render, e.g. `new Date(row.createdAt).toLocaleString()` — exact format is not asserted by the regex (`.* on .*`), any non-empty rendered date string satisfies it.

Table columns (left to right): Action, Entity type, Actor (staff name, or "System" if `actorId` is null), Timestamp, Source (badge: `rpc`/`edge`/`client`/`trigger`).

Infinite scroll: page size 50 (locked, D-04). Use a sentinel row + `IntersectionObserver` at the bottom of the table body (no existing pattern to copy — this is the first `useInfiniteQuery` usage in the codebase per RESEARCH.md) OR a manual "Load more" button — **default recommendation: manual "Load more" button** (simpler, no observer wiring, no risk of double-fetch races, and matches the codebase's general preference for explicit user-triggered actions over implicit scroll-triggered fetches seen elsewhere). Button label: "Load more entries". Disabled + shows a spinner while `isFetchingNextPage`.

### E. `AuditRoute` guard (`src/app/audit-route.tsx`, NEW)

Copy `RbacRoute`/`ReportsRoute` shape exactly, but ALSO fire the toast on redirect (this is the one guard in the app that needs a visible message — the others redirect silently):
```tsx
export function AuditRoute({ children }: { children: ReactNode }) {
  const { can } = usePermissions();
  if (!can('view_audit_log')) {
    toast.error('This page is restricted to managers and admins.');
    return <Navigate to="/home" replace />;
  }
  return <>{children}</>;
}
```
Requires a new `'view_audit_log'` `StaffAction` in `rbac.ts` (see item F below — this also fixes the HomeDashboard tile's gating mechanism in the same stroke, per Pitfall 8 in RESEARCH.md).

### F. HomeDashboard `/audit` tile — gating fix (not a new tile)

The tile already exists (`data-testid="home-tile-audit"`, label "Audit Log", `ClipboardList` icon) — do not create a duplicate or restyle it. Only change: swap `visibleToRoles: ['manager', 'admin']` for `requiredAction: 'view_audit_log'` (add this action to `MANAGER_EXTRA`/`ADMIN_EXTRA` in `rbac.ts` and to the `role_permissions` seed, matching every other manager+-gated tile's pattern from Phase 12/13). This makes the tile consistent with `Reports`/`Inventory`/`Settings`/`Kitchen Prep`/`Waitlist`/`Roles & Permissions`, all of which use `requiredAction` + `usePermissions().can(...)`, and gives it the same gray-lock-icon-when-denied visual treatment those tiles already have (no new visual design needed — inherited from the existing gated-tile pattern).

### G. `StaffDashboard` — "Force PIN Change" trigger (NEW UI surface, D-01a)

Add a new action to the existing per-row actions column (`src/widgets/StaffDashboard/StaffDashboard.tsx`, the `id: 'actions'` column), alongside "Clock In"/"Clock Out":
- Button: `variant="outline"` `size="sm"`, label **"Force PIN Change"**, gated via the same `ProtectedAction` wrapper pattern already used for Clock In/Clock Out (action `manage_staff` or a new dedicated `force_pin_change` `StaffAction` — planner's call, either is consistent with existing RBAC granularity).
- Icon: `KeyRound` (lucide-react) prefix, matching the icon-plus-label pattern used elsewhere in this column is optional (existing Clock In/Out buttons are text-only) — text-only is acceptable and consistent, do not force an icon if it clutters the row on narrow viewports.
- Clicking opens an `AlertDialog` (reuse `ManagerPinDialog`'s visual shell: `AlertDialogContent` → `AlertDialogHeader` → `AlertDialogTitle`/`AlertDialogDescription` → `AlertDialogFooter`) with the exact copy specified in the Copywriting Contract above (title includes staff name, body explains consequence, confirm button restates the verb).
- On confirm: calls the new `force_pin_change` RPC (backend scope, not this UI-SPEC's concern) and shows a success toast: **"{staffName}'s PIN will be changed on next login."**
- Row visual state after triggering: no persistent "pending" badge is required for this phase (out of scope per D-01a's build-minimum-viable framing) — a toast confirmation is sufficient; do not design a status column/badge unless the planner decides `mustChangePin` needs to be visible in the table (optional enhancement, not required by any locked decision or E2E test).

### H. Forced-PIN-change screen (staff-side, NEW UI surface, D-01a)

Lives inside `PINLoginForm.tsx`'s existing `Phase` state machine (`'pin' | 'opening_cash'` → extend to `'pin' | 'forced_pin_change' | 'opening_cash'`). Insert the check for `mustChangePin` immediately after successful `supabase.auth.signInWithPassword` (before the existing-shift lookup) — if `selectedStaff.mustChangePin === true`, set phase to `'forced_pin_change'` and block all further login progress (no existing-shift lookup, no opening-cash prompt) until the staff member successfully sets a new PIN.

Visual composition — reuse the exact same outer shell as the `'pin'` phase (avatar-initial circle + staff name + role, `flex flex-col gap-6 w-full max-w-xs` container) so the transition between phases feels continuous, not like a different screen:

```
[avatar circle + name + role — UNCHANGED from existing 'pin' phase]

Heading: "Set a new PIN"                         (Heading, 20px semibold)
Body: "A manager has required you to change      (Body, 14px regular, text-muted-foreground)
your PIN before continuing. Enter a new 4–6
digit PIN, then confirm it."

[PINKeypad #1 — label="New PIN", value=newPin, maxLength=4 or 6 matching
 existing PIN length convention]
  ↓ onComplete fires when maxLength reached
[PINKeypad #2 — label="Confirm new PIN", value=confirmPin, same maxLength]
  ↓ onComplete: compare newPin === confirmPin
    - mismatch → error "PINs don't match. Try again." (reuse PINKeypad's built-in
      error prop + red border, exact same visual as existing login error state),
      reset both fields to phase start
    - match but same as current PIN → error "Choose a PIN different from your
      current one."
    - match and different → call force_pin_change-clearing RPC/mutation
      (backend scope), on success: proceed to existing-shift lookup /
      opening-cash flow exactly as the normal 'pin' phase would have

[NO "Not you? Go back" link during this phase — the staff member is
 mid-mandatory-flow and must complete it, not abandon it. This is an
 intentional deviation from the 'pin' phase's UX, since backing out would
 leave mustChangePin=true unresolved.]
```

Do not build this as a separate route/page — it is a phase within the existing login widget, matching the `'opening_cash'` phase's precedent of gating progress within the same component rather than navigating away.

Two-step (enter, then confirm) is the required pattern — do not collapse to a single PIN entry, since a typo'd new PIN with no confirmation step would lock the staff member out on next login with no recovery path in scope for this phase.
