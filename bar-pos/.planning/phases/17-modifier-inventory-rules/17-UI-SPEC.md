# Phase 17 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope Note

This phase is a **small admin-CRUD extension**, not a new screen. It adds an "Ingredient rules" editor per modifier row inside the existing `CatalogModifiersTab.tsx` (`src/features/manage-products/ui/CatalogModifiersTab.tsx`), modeled directly on `RecipeEditorTab.tsx`'s ingredient-row-list pattern. All visual tokens below are **inherited from the existing codebase design system** (already in production, verified by direct file read) — nothing net-new is introduced. Source: `17-RESEARCH.md` Architecture Patterns / Recommended Project Structure; `components.json`; `tailwind.config.ts`; `src/app/globals.css`.

**Focal point:** the ingredient row list is the primary content of the dialog; the "Save rules" button is the secondary action anchor at the bottom-right, matching the existing `RecipeEditorTab.tsx` / `ModifierDialog` footer-action pattern.

| xl | 32px | Not used in this phase's scope |

Exceptions: none. The new "Ingredient rules" dialog is a modal (`Dialog` / `DialogContent`, `max-w-md` per `ModifierDialog`'s existing pattern extended to `sm:max-w-lg` if row count needs more horizontal room) — no non-4px-multiple values are needed since it clones the existing row-list layout verbatim.

## Typography

Inherited — no new type roles introduced. This phase reuses body text, labels, and the existing dialog title role exactly as `CatalogModifiersTab.tsx` and `RecipeEditorTab.tsx` already do.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px (`text-sm`) | 400 (regular) | 1.5 |
| Label | 14px (`text-sm`, `font-medium`) | 500 (medium — matches `FormField` label + `p.font-medium` row titles) | 1.5 |
| Heading (dialog title) | 16px (`text-base`) | 500 (medium — `font-medium`, matches the actual `DialogTitle` implementation at `src/shared/ui/dialog.tsx:110`: `font-heading text-base leading-none font-medium`) | 1.2 |
| Muted/helper text | 12px (`text-xs`, `text-pos-muted` / `text-muted-foreground`) | 400 (regular) | 1.5 |

Note: this collapses to exactly 2 font weights across the 4 declared roles/3 declared sizes (12/14/16px): regular (400) for Body/Muted, medium (500) for Label/Heading — satisfying the "3-4 sizes, max 2 weights" ceiling. The Heading row was corrected against the actual `DialogTitle` source (`src/shared/ui/dialog.tsx:110`) rather than shadcn's un-customized default; no file in this phase's scope (`CatalogModifiersTab.tsx`, `RecipeEditorTab.tsx`) uses `text-lg` or `font-semibold`.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA (per-row trigger, added next to Edit/Delete) | "Ingredient rules" (icon-only acceptable with `sr-only` text, matching Edit/Delete pattern — icon suggestion: lucide `FlaskConical` or `Beaker`; fallback to text label if no suitable icon: "Rules") |
| Primary CTA (dialog save button) | "Save rules" (mirrors `RecipeEditorTab.tsx`'s "Save recipe" label pattern — verb + noun, not generic "Save") |
| Empty state heading | None needed (no dedicated heading role in this compact dialog) |
| Empty state body | "No ingredient rules yet" (mirrors `RecipeEditorTab.tsx:154`'s exact phrasing "No recipe yet" — same tone, same italic muted style: `text-pos-muted text-sm italic`) |
| Error state | "Could not load ingredients: {message}" for load failures (mirrors `CatalogModifiersTab.tsx:44`'s exact pattern `text-destructive text-sm`); on save failure, `toast.error(r.error.message)` per existing `Result<T>` error surfacing convention — no separate inline error banner needed, matches existing mutation pattern in this file |
| Add-row action | "+ Add ingredient" (verbatim reuse of `RecipeEditorTab.tsx:205`'s exact label — do not invent a new label like "Add rule row") |
| Row remove action | `aria-label="Remove ingredient rule row"` on the trash icon button (adapted from `RecipeEditorTab.tsx:187`'s `"Remove ingredient row"` to disambiguate from the recipe editor's identical control when both appear in dev tools/tests) |
| Delta field label | "Delta" with hint text "Positive adds usage (e.g. extra cheese). Negative reduces usage (e.g. no ice)." (`FormField`-style hint, mirrors the existing `MoneyInput` hint pattern at `CatalogModifiersTab.tsx:254` — this hint is load-bearing since D-02's signed-delta semantics are non-obvious to an admin filling the form) |
| Destructive confirmation | This phase adds **no new destructive-with-confirmation flow**. Removing an ingredient-rule row within the dialog is a local (unsaved) edit, matching `RecipeEditorTab.tsx`'s `REMOVE_ROW` behavior (no confirm dialog — same as recipe rows, reversible via "Discard changes" until saved). The existing modifier-delete `ConfirmDialog` (title "Delete modifier?", already in `CatalogModifiersTab.tsx:150`) is unchanged and out of scope for this phase. |
| Save button loading state | "Saving…" (verbatim reuse of `RecipeEditorTab.tsx` save-button pattern via `isSaving` boolean, mirrors `ModifierDialogForm`'s existing `submitting ? 'Saving…' : 'Save'`) |

## States Checklist (for executor)

- [ ] Loading: `LoadingSpinner` centered, matches `RecipeEditorTab.tsx:137-143` exactly
- [ ] Empty (0 rules): italic muted "No ingredient rules yet"
- [ ] Populated (N rules): row list, each row = ingredient autocomplete + signed numeric input + remove button
- [ ] Dirty/clean: Save button disabled until a change is made (mirrors `RecipeEditorTab.tsx`'s `disabled={!state.isDirty || isSaving}`)
- [ ] Saving: button shows `LoadingSpinner` + "Saving…" text, inputs disabled
- [ ] Save success: `toast.success('Ingredient rules saved')` (mirrors `toast.success('Modifier saved')` pattern) + dialog stays open or closes per planner's discretion (existing `RecipeEditorTab` stays on the tab; this is a dialog, so close-on-save is the closer analog to `ModifierDialog`'s existing behavior)
- [ ] Save failure: `toast.error(r.error.message)`, dialog stays open, no data loss
- [ ] Negative delta entry (e.g. `-1`): must round-trip correctly — this is the explicit regression case from Pitfall 3 in `17-RESEARCH.md`; executor must NOT use `MoneyInput` for this field
