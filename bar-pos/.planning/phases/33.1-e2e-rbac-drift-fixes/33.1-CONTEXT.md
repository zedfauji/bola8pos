<decisions>
## Implementation Decisions

### Root Cause 1 — Toast stacking (`e2e/06-transfer.spec.ts` T4/T5)
- **D-01:** Toast stacking is correct, intended UX — confirmed by user, not a defect. Fix is test-side only: scope `getByText` assertions to exact per-tab toast text (mirrors this file's existing `Table 5`/`Table 12` exact-match pattern). No change to `App.tsx`'s `<Toaster>` or any `toast.success()` call site.
- **D-02:** Do NOT add a stable toast `id` for dedup/replace — that would be an app-side UX behavior change, explicitly rejected.
- **D-03:** After the T4 fix, re-run `e2e/06-transfer.spec.ts` in isolation (T4 + T5) twice to confirm T5 clears as a downstream cascade artifact, not an independent bug (per research's Pitfall 3). No separate root-cause investigation for T5 unless it still fails after T4 is green.

### Root Cause 2 — Aria-label collision (`e2e/09-rbac.spec.ts` T7)
- **D-04:** Fix `src/widgets/OrderPanel/ActiveTabSelector.tsx`'s Split-bill button: drop the customer-name interpolation, use static `aria-label="Split bill"` (matches sibling buttons' static-label convention in the same file). This is the file that actually renders on `/pos` — confirmed via `pos/index.tsx` import trace.
- **D-05 (scope expansion beyond research's baseline recommendation):** Delete `src/widgets/OrderPanel/OrderPanel.tsx` entirely, plus its re-export in `src/widgets/OrderPanel/index.ts`. Confirmed dead code: superseded by the `ActiveTabSelector`/`CartPanel` split (FSD one-concern-per-file refactor, git history shows `OrderPanel.tsx` predates the split and was never removed when `pos/index.tsx` switched to composing `ActiveTabSelector` + `CartPanel` directly). No import anywhere in `src/` references the `OrderPanel` name outside its own barrel export. User chose deletion over research's original lower-risk suggestion (mirror the 1-line aria-label fix into the dead file) — since the file is provably unused, remove it rather than patch a bug nobody can hit.
- **D-06:** No test file changes needed for Root Cause 2 — T7's own `test.skip()` escape hatch already handles the "feature not implemented" case correctly once the aria-label collision is gone.

### Root Cause 3 — PermissionMatrix drift (`e2e/09-rbac.spec.ts` T-RP-01/T-RP-02)
- **D-07:** Test-side only, per research: update `T-RP-01`'s expected switch count from `88` to `96` (and its stale "22 rows" comment to "24 rows"); add `exact: true` to `T-RP-02`'s `getByRole('switch', { name: 'Kitchen can view_kds' })` locator so it stops substring-matching `view_kds_bar`. No change to `rbac.ts`, `PermissionMatrix.tsx`, or any seed/RBAC data — the app and its unit test (`PermissionMatrix.test.tsx`) are already correct at 24/96.

### Documentation
- **D-08:** Once this phase closes, annotate the `COMPONENT-04` row in `.planning/REQUIREMENTS.md`'s traceability table (do not duplicate the row) noting this phase is what makes its "verified by existing E2E specs... passing unchanged" clause true.

### Claude's Discretion
- Exact wording of updated comments in `e2e/06-transfer.spec.ts` and `e2e/09-rbac.spec.ts` (e.g., the "24 rows x 4 roles" comment phrasing).
- Whether to remove `OrderPanel.tsx`'s barrel export line before or as part of the same commit as the file deletion — both are part of D-05, just sequencing.
