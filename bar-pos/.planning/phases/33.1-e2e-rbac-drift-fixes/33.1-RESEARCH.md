# Phase 33.1: E2E Test/RBAC Drift Fixes - Research

**Researched:** 2026-07-13
**Domain:** Playwright E2E test/locator drift + a dead-code aria-label collision (no new libraries, no backend/DB changes)
**Confidence:** HIGH (all three root causes independently confirmed by direct source read + grep, not inferred)

## Summary

This phase is not "build something new" — it is three small, unrelated, already-root-caused bugs that Phase 33's verifier surfaced and explicitly kept out of scope. All three were re-traced from scratch in this research session and each root cause is now **fully confirmed at the code level**, not merely hypothesized:

1. **`e2e/06-transfer.spec.ts` T4/T5** — `sonner`'s default toast lifetime is 4000ms (confirmed in installed `node_modules/sonner`) and the toast stacking is *intentional library behavior*, not a production bug. T4 fires two "Tab opened for X" toasts back-to-back and then asserts on the generic regex `getByText(/tab opened/i)`, which now matches 2 stacked toasts (strict-mode violation). This is a **test-locator specificity bug**, not an App.tsx defect — the fix is scoping each assertion to its own toast text (a pattern the test already uses elsewhere for `Table 5`/`Table 12`). T5's cascading failure is very likely a downstream artifact of T4's throw within the same worker (this repo runs `workers: 1`) and should be re-verified in isolation once T4 is fixed — it does not need its own root-cause fix.
2. **`e2e/09-rbac.spec.ts` T7** — There is **no "Delete Tab" feature anywhere in `src/`** (confirmed via grep — `delete_tab` exists only as an RBAC action constant in `rbac.ts`, never wired to any button). T7's own code anticipates this ("UI not implemented — EXPECTED FAIL") and has a `test.skip()` escape hatch. That hatch is broken because `ActiveTabSelector.tsx`'s "Split bill" button interpolates the raw customer name into its `aria-label` (`Split tab ${currentTab.customerName}`), and the seeded tab name "RBAC Delete Tab" makes that label literally contain the substring "Delete Tab", so `getByRole('button', {name: /delete tab/i})` incorrectly resolves to the Split-bill button instead of "nothing." **Only `ActiveTabSelector.tsx` matters at runtime** — `OrderPanel.tsx`'s copy of this same pattern is dead code (its `OrderPanel` export is never imported anywhere in `src/`; `pos/index.tsx` composes `ActiveTabSelector` + `CartPanel` directly). The correct fix is app-side: drop the customer-name interpolation and use a static `aria-label="Split bill"` (matching the sibling buttons' own static-label convention in the same file) — zero test changes required, and confirmed safe because the only E2E spec that interacts with this button (`e2e/34-split-bill.spec.ts`) already uses `getByTestId('split-bill-button')`, never the aria-label text.
3. **`e2e/09-rbac.spec.ts` T-RP-01/T-RP-02** — `STAFF_ACTIONS` in `rbac.ts` has grown to **24** entries (not 22, as the 2026-04-27-era test comment assumes); `PermissionMatrix.test.tsx` (unit test, already updated in a later phase) independently confirms `24 * 4 = 96` switches. The app is correct; only the E2E test's hardcoded `88` and its non-exact `name: 'Kitchen can view_kds'` locator (which substring-matches the newer `Kitchen can view_kds_bar` switch) are stale. Both are pure test-side updates.

**Primary recommendation:** Fix (1) as a test-locator change only (no App.tsx edit), fix (2) as a 1-line app-side aria-label change in `ActiveTabSelector.tsx` (optionally mirrored in the dead `OrderPanel.tsx` for consistency, at zero risk), and fix (3) as a 2-line test-side update (count + `exact: true`). No new dependencies, no DB/RPC changes, no behavior change to any payment-critical flow.

## Root Cause 1 — `e2e/06-transfer.spec.ts` T4/T5 (toast stacking)

**File chain:** `src/app/App.tsx` (`<Toaster richColors position="top-right" />`) → `src/features/open-tab/ui/OpenTabDialog.tsx:104` (`toast.success(\`Tab opened for ${result.data.customerName}\`)`) → `src/features/transfer-tab/ui/TransferTabDialog.tsx:44` (`toast.success('Tab transferred successfully.')`).

**Confirmed facts (read directly from installed `node_modules/sonner/dist/index.mjs`, sonner `^2.0.7` per `package.json`):**
- `TOAST_LIFETIME = 4000` (ms) — the default duration for every `toast.success()` call unless overridden per-call or via `<Toaster duration={...}>`. `[VERIFIED: node_modules/sonner installed source]`
- `VISIBLE_TOASTS_AMOUNT = 3` — sonner's default `visibleToasts`, i.e. it is designed to show up to 3 stacked toasts at once; excess toasts are pushed behind visually but remain mounted in the DOM until their own timer expires. `[VERIFIED: node_modules/sonner installed source]`
- Neither `toast.success()` call in `OpenTabDialog.tsx`/`TransferTabDialog.tsx` passes an `id`, so sonner treats every call as a brand-new toast (no dedup/replace).

**Why T4 fails:** T4 opens Tab A, then *immediately* (no wait beyond the `tab opened` toast becoming visible, i.e. no wait for it to *disappear*) opens Tab B. Toast A ("Tab opened for Transfer Conflict A") is very likely still on-screen (4s lifetime) when Toast B ("Tab opened for Transfer Conflict B") appears and the test asserts `getByText(/tab opened/i)`. Both toasts satisfy the regex → Playwright strict-mode violation (locator resolves to 2 elements) → test throws before `logout(page)` runs.

**Why T5 (likely) fails:** `playwright.config.ts` has `workers: 1`, `fullyParallel: false`, `retries: 1`. T5 does not itself re-use T4's `page` (each test still gets its own Playwright `page`/context by default), but running in the same single worker process means a hard throw partway through T4 (before its `logout()`/cleanup) can leave the *server-side* app state (an open dialog's mutation in-flight, a stray created tab) or, per the verifier's report, occasionally the browser process itself in a bad state for the next test in-process. This is a downstream symptom, not an independent root cause — **no separate fix is prescribed for T5**; the plan should simply re-run `e2e/06-transfer.spec.ts` in isolation after the T4 fix and confirm 5/5 pass before closing this bug.

**Is this an app bug or a test bug?** Toast stacking on rapid consecutive actions is *correct, intentional* sonner/toast-library behavior — real users legitimately see 2 "tab opened" toasts if they create two tabs quickly, and that's desirable feedback, not a defect. The defect is that the **test** uses an ambiguous generic regex where two toasts can legitimately coexist. Every other passing test in this same file (T1–T3) opens only one tab per assertion, so they never hit this collision — T4/T5 are the only tests that create 2+ tabs back-to-back.

**Recommended fix (test-side, root cause, zero prod risk):** Scope each `getByText` toast assertion in T4 to the exact per-tab text, reusing the pattern the same file already uses for exact `Table 5`/`Table 12` checks — e.g. `getByText('Tab opened for Transfer Conflict A')` / `getByText('Tab opened for Transfer Conflict B')` instead of the shared `/tab opened/i`. This requires **no change to `App.tsx` or any `toast.success()` call site**.

**Optional app-side alternative (not required, do not do both):** Give the "Tab opened" toast a stable `id` per dialog instance (e.g. `toast.success(msg, { id: 'tab-opened' })`) so a second rapid call replaces rather than stacks. Sonner supports this natively (id-based toast dedup/replace is the documented "don't hand-roll a toast queue" pattern) `[ASSUMED — from training knowledge of sonner's public API, not re-verified against sonner's docs site in this session; the `id` option is present in the installed source's toast function signature but its replace-on-duplicate-id semantics were not independently exercised here]`. This is a real (minor) UX behavior change outside the phase's stated "fix pre-existing drift, don't change behavior" framing — **skip this unless the test-side fix alone is judged insufficient.**

## Root Cause 2 — `e2e/09-rbac.spec.ts` T7 (aria-label collision)

**File:** `src/widgets/OrderPanel/ActiveTabSelector.tsx` line 123:
```tsx
aria-label={`Split tab ${currentTab.customerName}`}
```
and the byte-identical pattern in `src/widgets/OrderPanel/OrderPanel.tsx` line 146.

**Confirmed facts:**
- `grep -r "[Dd]elete [Tt]ab" src/` returns **zero matches** — there is no "Delete Tab" UI anywhere in the app. `delete_tab` exists only as a `StaffAction` string literal in `src/shared/lib/rbac.ts` (admin-only, via `ADMIN_EXTRA`), never wired to a button, dialog, or handler. `[VERIFIED: grep across src/]`
- `src/widgets/OrderPanel/OrderPanel.tsx`'s exported `OrderPanel` component is **not imported anywhere in `src/`** (`grep "from '@widgets/OrderPanel'"` only finds `pos/index.tsx` importing `{ ActiveTabSelector, CartPanel }`, never `OrderPanel` itself). It is dead code with respect to the live render tree. `[VERIFIED: grep across src/]`
- `e2e/34-split-bill.spec.ts` (the spec that actually exercises the Split-bill button) selects it exclusively via `page.getByTestId('split-bill-button')` — never by aria-label/accessible-name regex. `[VERIFIED: grep of e2e/34-split-bill.spec.ts]`

**Why T7 fails:** T7's own code already anticipates the delete-tab feature might not exist:
```ts
const deleteBtn = page.getByRole('button', { name: /delete tab/i });
const hasDelete = await deleteBtn.isVisible({ timeout: 5_000 }).catch(() => false);
if (!hasDelete) { test.skip(true, 'UI not implemented — EXPECTED FAIL: ...'); return; }
```
Because the seeded tab is named `'RBAC Delete Tab'`, `ActiveTabSelector.tsx`'s Split-bill button ends up with `aria-label="Split tab RBAC Delete Tab"`, which contains the substring "Delete Tab" and satisfies `/delete tab/i`. So `hasDelete` resolves `true` (for the *wrong* button), the test proceeds to `.click()` the Split-bill button (opening `SplitTabSheet`, not deleting anything), never finds a `tab deleted` toast, and times out/fails — instead of cleanly hitting its own intended skip path.

**Recommended fix (app-side, root cause, in the phase's named file):** Remove the customer-name interpolation from the Split-bill button's `aria-label` in `ActiveTabSelector.tsx` — there is only ever one Split-bill button visible at a time (it renders only for the single active tab), so the name adds no disambiguation value, and dropping it matches the static-label convention already used by every sibling button in the same component (`aria-label="Transfer tab"`, `aria-label="New tab"`, `aria-label="Switch tab"`). Confirmed safe: no E2E spec depends on the interpolated text (only `data-testid="split-bill-button"` is used). This alone restores T7's intended skip-and-pass behavior with **zero test file changes**.

**Secondary, optional fix:** apply the identical one-line change to `OrderPanel.tsx`'s copy for consistency (it's dead code today, so this doesn't affect any currently-passing test, but prevents the same class of bug from resurfacing silently if `OrderPanel.tsx` is ever wired back up). Low cost, recommended but not blocking.

## Root Cause 3 — `e2e/09-rbac.spec.ts` T-RP-01/T-RP-02 (PermissionMatrix drift)

**File:** `src/shared/lib/rbac.ts` — `STAFF_ACTIONS` (source of truth), consumed unmodified by `src/widgets/RBACDashboard/PermissionMatrix.tsx`.

**Confirmed facts:**
- `STAFF_ACTIONS` currently has **24** entries (counted directly from the array literal): `create_order, view_own_tabs, view_all_tabs, start_pool_timer, stop_pool_timer, clock_in, clock_out, close_tab, void_order, view_reports, adjust_inventory, manage_products, manage_staff, manage_settings, delete_tab, view_all_shifts, manage_caja, transfer_tab, view_kds, view_kds_bar, process_refund, produce_prep_batch, manage_waitlist, view_audit_log`. `[VERIFIED: direct read of src/shared/lib/rbac.ts]`
- `src/widgets/RBACDashboard/PermissionMatrix.test.tsx` (unit test, Vitest) **already asserts 24 rows / 96 switches** (`expect(STAFF_ACTIONS).toHaveLength(24)`, `expect(switches).toHaveLength(24 * 4)`) — this is a later, correct, already-passing unit test that independently corroborates the count. `[VERIFIED: direct read of PermissionMatrix.test.tsx]`
- `e2e/09-rbac.spec.ts` T-RP-01 still asserts `toHaveCount(88)` (`22 * 4`, per its own stale comment "STAFF_ACTIONS has 22 items") — this is the only place in the codebase still using the old count.
- `PermissionMatrix.tsx`'s switch `aria-label` template is `` `${ROLE_LABELS[role]} can ${action}` ``. Because `view_kds` and `view_kds_bar` are both real, distinct actions and Playwright's `getByRole(role, {name})` does a **case-insensitive substring match by default** (not exact) when `name` is a plain string, the un-anchored locator `getByRole('switch', { name: 'Kitchen can view_kds' })` in T-RP-02 matches both `"Kitchen can view_kds"` and `"Kitchen can view_kds_bar"` → strict-mode violation.

**Conclusion:** the app is correct and already has an up-to-date unit test proving it; only the E2E spec is stale. **Both fixes are test-side only:**
1. T-RP-01: update the expected count from `88` to `96` (and its comment from `22 rows x 4 roles` to `24 rows x 4 roles`).
2. T-RP-02: add `exact: true` to the `getByRole('switch', { name: 'Kitchen can view_kds', exact: true })` locator (or use an anchored regex `/^Kitchen can view_kds$/`) so it no longer substring-matches `view_kds_bar`.

No changes needed to `PermissionMatrix.tsx`, `rbac.ts`, or any seed/RBAC data.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Distinguishing 2 stacked toasts in a Playwright assertion | A custom "wait for all toasts to clear" helper/sleep | Exact-text `getByText('<specific toast text>')`, matching the pattern already used elsewhere in the same spec file for `Table 5`/`Table 12` | Sleeping/polling for toast dismissal is flaky and slow; exact-text scoping is instant and matches existing conventions |
| Disambiguating `view_kds` vs `view_kds_bar` switch locators | A rename of the action strings or the aria-label template | `exact: true` on the existing `getByRole('switch', {name})` locator | Playwright's own `exact` option is built for this; renaming a public RBAC action string (`view_kds`) has a much larger blast radius (DB `role_permissions` rows, other RBAC checks) for zero benefit |

**Key insight:** all three bugs are locator/test-string precision issues (Playwright's default substring matching plus a raw string interpolated into an `aria-label`) — the fix surface is a handful of one-line, surgical edits, not new abstractions or helpers.

## Common Pitfalls

### Pitfall 1: Assuming the ROADMAP's named file is always the fix location
**What goes wrong:** Planning tasks purely off the ROADMAP bullet text ("Toast-stacking timing bug in `src/app/App.tsx`'s Toaster", "`PermissionMatrix.tsx` switch-count/action-set drift") would lead a task-writer to touch `App.tsx` and `PermissionMatrix.tsx` — files that, on inspection, are either correct-as-is or where the *test* is what's stale.
**Why it happens:** The ROADMAP entry was written by the verifier as a symptom description ("this app file's *area*"), not a pre-verified fix location.
**How to avoid:** Task-level plans should target the actual files identified in this research (`ActiveTabSelector.tsx` for the real app-side fix; `e2e/06-transfer.spec.ts` and `e2e/09-rbac.spec.ts` for the two test-side fixes), with an explicit note that `App.tsx` and `PermissionMatrix.tsx` require **no code change**.
**Warning signs:** A diff that touches `App.tsx`'s `<Toaster>` props or `PermissionMatrix.tsx`'s aria-label template when the E2E-spec-only fix already makes the target tests pass — that's scope creep on a phase explicitly meant to be small and surgical.

### Pitfall 2: "Fixing" the aria-label collision by making the test locator more specific instead of fixing the app
**What goes wrong:** It's tempting to fix T7 by narrowing `getByRole('button', {name: /delete tab/i})` to something that doesn't match "Split tab RBAC Delete Tab" (e.g. `exact: true` with a literal "Delete tab" string). This "works" for this one seeded tab name, but the underlying fragility (any button whose aria-label embeds a raw, tester-controlled string) remains and will resurface with a different seeded name in a future test.
**Why it happens:** It looks like a smaller diff (test-only) and mirrors the fix pattern used for Root Cause 3.
**How to avoid:** For Root Cause 2 specifically, fix the app's aria-label (drop the interpolation) — it is both the smaller diff (1 line, one file, confirmed zero test dependents) and the actually-durable fix. Root Cause 3's `exact: true` fix is fine as test-side because there the *app's* aria-label template (`"{role} can {action}"`) is fine and it's the STAFF_ACTIONS *count/set*, not the template, that changed.
**Warning signs:** A fix that only patches the exact string `'RBAC Delete Tab'` case rather than removing the general hazard.

### Pitfall 3: Treating T5's failure as requiring its own root-cause investigation
**What goes wrong:** Spending planning effort tracing "Target page, context or browser has been closed" as if it were a fourth, independent bug.
**Why it happens:** It's a different error message than T4's strict-mode violation, so it looks unrelated.
**How to avoid:** Fix T4 first, then re-run `e2e/06-transfer.spec.ts` (both T4 and T5, in isolation, twice per this session's own verification methodology) before spending further investigation time. If T5 still fails after T4 is green, escalate then — not before.
**Warning signs:** A plan task that tries to patch pool-session/transfer-tab RPC code to "fix T5" without first confirming T5 still fails once T4 is fixed.
