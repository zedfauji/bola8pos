## Warnings

### WR-01: `navigationTimeout` left unbounded in default (non-fast) Playwright mode — same hang class the sibling fix just closed

**File:** `playwright.config.ts:41-42`
**Issue:** This phase's `5f7cbc7` commit set `actionTimeout: fastE2e ? 10_000 : 15_000` (previously `undefined`) specifically because an unbounded action timeout let a stale-element click silently consume a test's *entire* remaining timeout budget (120s) before its `.catch()` ever ran. `navigationTimeout` sits immediately below it and still has the exact same shape: `fastE2e ? 15_000 : undefined`. Any `page.goto()`/navigation that stalls in the default (non-`FAST_E2E`) run mode — which is the mode used for normal local/manual runs per this file's own `slowMo`/`testTimeout` branching — will poll for the full `testTimeout` (60s) instead of failing fast with a clear "navigation timeout" error, reproducing the same debugging pain the sibling fix was written to eliminate.
**Fix:**
```ts
navigationTimeout: fastE2e ? 15_000 : 30_000,
```

### WR-02: Duplicate-table-transfer guard check in T4 discards its result — asserts nothing

**File:** `e2e/06-transfer.spec.ts:141-146`
**Issue:**
```ts
await page
  .getByText(/already.*occupied|table.*taken|conflict|duplicate/i)
  .isVisible({ timeout: 4_000 })
  .catch(() => false);
```
The resolved boolean from `.isVisible()` is never captured or asserted against — the comment above ("both are acceptable, so this is a soft check only") documents the intent, but the code doesn't even do a soft check: it awaits a promise and throws the result away. If a future regression made the RPC's duplicate-table guard start silently corrupting state (e.g. two tabs pointing at the same occupied table with no error ever surfacing, when previously an error was shown), this test would still report green with zero evidence either way. This is meaningfully different from "accepting either outcome" — it accepts and *records* neither.
**Fix:** At minimum, log the observed state so the intent is verifiable when someone reads a test run's trace/report:
```ts
const conflictShown = await page
  .getByText(/already.*occupied|table.*taken|conflict|duplicate/i)
  .isVisible({ timeout: 4_000 })
  .catch(() => false);
test.info().annotations.push({
  type: 'note',
  description: `Duplicate-table transfer conflict toast shown: ${conflictShown}`,
});
```

## Info

### IN-01: "New Tab +" button's `aria-label` overrides its own visible text, creating a Label-in-Name mismatch

**File:** `src/widgets/OrderPanel/ActiveTabSelector.tsx:99-110`
**Issue:** The active-tab-state "New Tab +" button has visible text `New Tab +` but also sets `aria-label="New tab"` (line 106), which per accessible-name computation *replaces* the visible text as the button's accessible name rather than supplementing it — a WCAG 2.5.3 (Label in Name) violation, since a screen-reader/voice-control user hears "New tab" while sighted users see "New Tab +". This is inconsistent with the sibling icon-only "Transfer tab" button (where `aria-label` is required because there's no visible text) and with the equivalent "New Tab +" button in the empty-tab-state branch (lines 189-198), which has no `aria-label` at all and correctly derives its accessible name from the visible text + icon.
**Fix:** Drop the redundant `aria-label` on the text-bearing button to match the empty-state variant:
```tsx
<Button
  type="button"
  variant="outline"
  size="sm"
  onClick={() => {
    setOpenTabDialogOpen(true);
  }}
>
  <Plus className="mr-1 h-4 w-4" />
  New Tab +
</Button>
```

### IN-02: Redundant/tautological re-check in T-RP-05's negative branch

**File:** `e2e/09-rbac.spec.ts:359-366`
**Issue:** `refundVisible` is already established as `false` (via `.first().isVisible({ timeout: 3_000 }).catch(() => false)`) before entering the `else` branch. The branch then re-queries the same locator (`page.getByRole('button', { name: /refund/i })`, this time without `.first()` or a timeout) and asserts the negation is `true`. Since a zero-match locator's `.isVisible()` resolves to `false` (not a rejection), `stillNotVisible` is true by construction and this assertion cannot meaningfully fail independently of the branch condition that led here — it re-states rather than re-verifies.
**Fix:** Either delete the redundant re-check (the `if/else` structure already documents both accepted outcomes), or replace it with a `toHaveCount(0)` assertion against a scoped locator if the intent is to also guard against refund buttons appearing asynchronously after the initial check.
