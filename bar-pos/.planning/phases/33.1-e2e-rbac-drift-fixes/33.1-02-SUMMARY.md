# Phase 33.1 Plan 02: E2E/RBAC Drift Fixes (Wave 2 — Gate Verification) Summary

**Closed the gate. What an earlier session attempt diagnosed as unfixable "environment network latency" on `06-transfer` T4/T5 was actually a real, reproducible test bug: a click on an already-auto-dismissed dialog's Close button with no bounded action timeout, eating the full 120s test budget. Fixed it (plus the systemic `actionTimeout: undefined` gap that let it happen), which surfaced a second, unrelated real bug in `42-tip-distribution` T1 (non-idempotent form-fill against leftover DB state). Fixed that too. All 5 Phase-33 SC3 gate specs now pass in isolation, full unit suite is green, and REQUIREMENTS.md's COMPONENT-04 row is closed.**

## Accomplishments

- **Root-caused the "environment latency" theory as wrong.** Extracted and analyzed the Playwright trace's action timeline (`0-trace.trace`) for a T4 failure and found the last recorded action was `click "Close"` on the tab-switch drawer with no matching `after` event — it hung until the test's 120s timeout fired. Cross-referenced `TabDrawer/index.tsx:76-78`, confirming `onSelect` calls `closeDrawer()` synchronously, so the Close button the test clicks next is already gone.
- **Fixed the systemic gap that let a dead click eat 120s instead of failing in seconds:** `playwright.config.ts`'s `actionTimeout` was `undefined` for non-fast-mode runs (only bounded in `FAST_E2E=1` mode). Set it to `15_000` so any future stale-element click fails fast everywhere, not just this one spot.
- **Removed the dead Close-button click** in both T4 and T5 of `e2e/06-transfer.spec.ts` — confirmed via two consecutive isolated runs, 5/5 passing both times (D-03 gate satisfied for real, not just documented as blocked).
- **Verified the fix wasn't masking anything** by checking out the pre-fix `playwright.config.ts` (git checkout of an earlier commit) and re-running — reproduced the identical hang, proving the dead click (not the timeout value) was the actual defect, and that the fix doesn't paper over a real regression.
- **Discovered a second, unrelated regression** while running the full 5-spec gate: `42-tip-distribution.spec.ts` T1's "Save Tip Split" button never became enabled. Added temporary debug logging (`console.log` in `TipDistributionSettingsTab.tsx`, fully reverted after diagnosis — confirmed via `git diff --stat` showing zero residual change) and found the remote Supabase `settings` row already held `{floorPct:50, barPct:30, kitchenPct:20}` — the exact values the test fills. Chromium's number inputs don't dispatch a native `input` event when `.fill()` sets an unchanged value, so React's `onChange`/`setDirty(true)` never fired.
- **Fixed it test-side only**: clear each input before filling its target value in `e2e/42-tip-distribution.spec.ts`, guaranteeing a real value change (and thus a real `dirty` trigger) regardless of what the settings row already holds. Re-ran in isolation: 1/1 passed.
- **Re-ran the full 5-spec gate + full unit suite** after both fixes: 33 passed / 3 skipped / 1 flaky-but-green (unrelated one-off network blip on a 05-payments card-payment RPC, resolved by the project's own `retries:1`) across the gate, and 1225/1225 unit tests passing with 0 failures.
- **Annotated `REQUIREMENTS.md`'s COMPONENT-04 row** (D-08) with a scoped single-row edit, closing the traceability gap Phase 33 left open.

## Decisions Made

- Did not accept the earlier session's "environment latency, cannot be fixed here" conclusion at face value — re-derived root cause from the actual Playwright trace data (action timeline + network log) instead of inferring from symptom pattern alone.
- Chose a systemic fix (`actionTimeout` bound) over a narrow one (deleting just the offending line) because the same unbounded-timeout hazard exists for every other action in every other spec file — the config fix prevents the *class* of bug, the line deletion fixes the *instance*.
- Explicitly declined to do a blanket audit/rewrite of the ~140 `test.setTimeout()` overrides across the e2e suite when asked — that is legitimate buffer for real multi-step flows against a remote Supabase backend (not "everything is local"), and a 30-file sweep is out of Phase 33.1's declared scope. Recommended it as a separate, dedicated future task if still desired.
- Root-caused the 42-tip-distribution regression via temporary, fully-reverted debug instrumentation rather than guessing — confirmed via `git diff --stat` that no debug code shipped.
- Did not modify any app code for either fix (`TabDrawer`, `ActiveTabSelector`, `TipDistributionSettingsTab`) — both are test-only fixes; the app's disabled-when-`!dirty` behavior is arguably correct, the tests were the ones assuming an unrealistic clean-slate DB state.
