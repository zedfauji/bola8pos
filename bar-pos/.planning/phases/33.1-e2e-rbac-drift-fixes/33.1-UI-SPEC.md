## Interaction & Accessibility Contract (the actual deliverable of this spec)

This is the load-bearing section for this phase. It exists to lock the exact strings the executor must produce and the exact matching precision the test fixes must use, so a plan/executor pass can't silently drift the fix (e.g. leaving the customer-name interpolation in, or using a loose regex where an exact match is required).

### D-04 — `ActiveTabSelector.tsx` Split-bill button `aria-label`

| Property | Contract |
|----------|----------|
| File | `src/widgets/OrderPanel/ActiveTabSelector.tsx` (line ~123) |
| Before | `aria-label={\`Split tab ${currentTab.customerName}\`}` (dynamic — collides with any seeded tab name containing "delete tab", e.g. `"RBAC Delete Tab"`) |
| After | `aria-label="Split bill"` (static string, exact) |
| Rationale | Matches the static-label convention of every sibling button in the same file: `aria-label="Transfer tab"`, `aria-label="New tab"` (see lines 94, 106). Only one Split-bill button renders at a time (gated on `currentTab.status === 'open'`), so no disambiguation value is lost by dropping the interpolation. |
| Visible button text | Unchanged: `Split bill` (the `<span>`/text content inside the button is NOT part of this fix — only the `aria-label` attribute) |
| `data-testid` | Unchanged: `"split-bill-button"` — confirmed the only locator `e2e/34-split-bill.spec.ts` depends on; do not touch |
| Do NOT touch | `src/app/App.tsx`'s `<Toaster>` props, `PermissionMatrix.tsx`'s aria-label template — both are correct as-is per research Pitfall 1 |

### Toast copy — unchanged, contract is test-side matching precision only

No `toast.success()` call site or its message string changes in this phase (per D-01/D-02 — confirmed intentional, not touching `App.tsx`'s `<Toaster>` or any toast copy). The existing messages remain the contract:

| Toast trigger | Exact copy (unchanged) |
|---------------|------------------------|
| Tab opened | `Tab opened for {customerName}` |
| Tab transferred | `Tab transferred successfully.` |

**Test-side matching contract (`e2e/06-transfer.spec.ts` T4):** replace the ambiguous shared-regex assertion `getByText(/tab opened/i)` — which legitimately matches 2 stacked toasts when two tabs are opened back-to-back — with exact per-tab text assertions: `getByText('Tab opened for Transfer Conflict A')` then `getByText('Tab opened for Transfer Conflict B')`. This mirrors the file's own existing exact-match convention for `Table 5`/`Table 12` (see lines 29/36 of that spec). Do not add a `sleep`/wait-for-toast-to-clear helper — exact-text scoping is the correct, non-flaky fix per research's "Don't Hand-Roll" table.

### Test-side locator precision contract (`e2e/09-rbac.spec.ts`)

| Test | Before | After |
|------|--------|-------|
| T-RP-01 | `expect(switches).toHaveCount(88)` + comment `22 rows × 4 roles = 88 switches` | `expect(switches).toHaveCount(96)` + comment `24 rows × 4 roles = 96 switches` (matches `STAFF_ACTIONS.length === 24`, already corroborated by `PermissionMatrix.test.tsx`) |
| T-RP-02 | `page.getByRole('switch', { name: 'Kitchen can view_kds' })` (substring-matches `view_kds_bar` too) | `page.getByRole('switch', { name: 'Kitchen can view_kds', exact: true })` |

No change to `PermissionMatrix.tsx`'s aria-label template (`` `${ROLE_LABELS[role]} can ${action}` ``) — it is correct; only the E2E locator's matching mode changes.

### T7's own escape hatch — unchanged

`e2e/09-rbac.spec.ts` T7 keeps its existing `test.skip()` fallback (`getByRole('button', { name: /delete tab/i })` → not visible → skip). No test-file edit needed for T7 (D-06) — fixing D-04's aria-label is sufficient to let this escape hatch resolve correctly (no accidental match against the Split-bill button).
