<objective>
Extend `PaymentForm.tsx` with the split-payment UI: a "Split payment" toggle that swaps the single method-selector for an inline stacked list of 2-4 payment rows (D-01), each with its own method / amount / tip / method-specific fields (D-03), a live remaining-balance box (D-07), and a submit path that calls `processSplitPayment` then displays one receipt per leg in sequence with the cash drawer opening once per checkout (D-09). The single-method (toggle-OFF) flow stays byte-for-byte unchanged (D-11 / SC-4).

Purpose: This is the operator-facing deliverable of Phase 18 — SC-3. Every visual/interaction decision is pinned by 18-UI-SPEC.md (inline stacked row cards, NOT accordion); the row-list state machine copies the `RecipeEditorTab` useReducer pattern; the money math reuses PaymentForm's existing `Math.round(x*100)/100` idiom and discount/tax `useMemo` chain unchanged.

Output: Extended `PaymentForm.tsx` + new split-mode RTL tests. Unit-tested with a mocked `processSplitPayment` — no live edge function needed here.
</objective>
