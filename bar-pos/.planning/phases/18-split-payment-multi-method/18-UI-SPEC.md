# Phase 18 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope:** This phase adds a "Split payment" toggle and repeatable row list inside the existing `src/widgets/PaymentModal/ui/PaymentForm.tsx`. No new screen, no new design-system components, no new registries. Every visual decision below is either (a) inherited directly from `PaymentForm.tsx`'s current single-payment markup, or (b) copied from the closest existing row-list precedent in this codebase, `src/features/manage-recipe/ui/RecipeEditorTab.tsx` (ingredient row add/remove pattern). This spec exists specifically to lock down the "Claude's Discretion" item in `18-CONTEXT.md`: *"Exact UI layout for adding/removing rows within `PaymentForm` (inline stacked rows vs. accordion, etc.)"* — resolved below as **inline stacked row cards**, not an accordion.

## Spacing Scale

Declared values (must be multiples of 4) — these are the values already live in `PaymentForm.tsx` and `RecipeEditorTab.tsx`; no new scale is introduced:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps (`gap-2` is actually 8px — see below; true 4px use is rare, e.g. `mt-0.5` on `AlertCircle`) |
| sm | 8px | Row-internal gaps (`gap-2`), compact stacking (`space-y-2`) |
| md | 16px | Default section spacing (`space-y-3`/`space-y-4`), card padding (`p-3`/`p-4`) |
| lg | 24px | Section-to-section spacing (`space-y-6` container) |
| xl | 32px | Not used inside `PaymentForm` — reserved for page-level layout only |
| 2xl | 48px | Not used inside `PaymentForm` |
| 3xl | 64px | Not used inside `PaymentForm` |

Exceptions:
- Icon-only row-remove buttons (`Trash2` in a ghost `POSButton`) use the project's standard **44px minimum touch target** (`POSButton` default `touchSize="default"` = `min-h-[44px]`), even though the icon itself is 16px (`h-4 w-4`). This matches the existing `RecipeEditorTab.tsx` remove-row button exactly — do not shrink below 44px regardless of visual density, per project touch-target convention (also documented in Phase 8's S6-12 paper-cuts work).
- Per-row method-selector buttons use `touchSize="large"` (56px), one size down from the single-payment flow's `touchSize="xl"` (72px) method buttons — necessary because up to 4 rows must fit on screen simultaneously without excessive scrolling. This is the one deliberate spacing/sizing exception for this phase.

## Typography

`PaymentForm.tsx` already establishes a working scale (`text-xs` / `text-sm` / `text-base` default / `text-lg`, weights `font-medium` / `font-semibold`). New split-mode elements reuse this scale exactly — no new sizes or weights introduced.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body / row values (amounts, inputs, helper text) | 14px (`text-sm`) | Regular (400, default/unset) | 1.5 |
| Label / row headings ("Payment 1", "Remaining to pay", section `h4`s) | 14px (`text-sm`) | Semibold (600, `font-semibold`) | 1.5 |
| Section heading (tab name, "Receipt", card total) | 18px (`text-lg`) | Semibold (600, `font-semibold`) | 1.2 |
| Fine print (helper/muted copy under toggle, row count hint) | 12px (`text-xs`) | Regular (400) | 1.5 |

Only 2 weights declared for **new** elements introduced by this phase: **Regular (400)** for values/body copy, **Semibold (600)** for labels/headings/totals/primary CTA. (Note: the existing unmodified sections of `PaymentForm.tsx` also use `font-medium` (500) on some `h4`s — that is pre-existing, out of scope, and not to be copied into new split-mode markup.)

## Color

Reuses the existing CSS variable palette (`src/app/*.css`, dark mode is default per `darkMode: 'media'` + system preference). No new colors introduced.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `var(--background)` (`oklch(0.145 0 0)` dark) | Page/modal background |
| Secondary (30%) | `var(--card)` / `var(--muted)` via `rounded-lg border p-3`/`p-4` boxes | Each split-payment row card, the remaining-balance box, the totals summary box — all use the existing bordered-box pattern already seen in `PaymentForm.tsx`'s totals section |
| Accent (10%) | `var(--pos-accent)` (green, `oklch(0.78 0.2 145)` dark) | Reserved for exactly two elements: (1) the primary submit button background (`bg-[var(--pos-accent)] text-black`, unchanged from single-payment flow, applies to "Process split payment" too), and (2) the "fully allocated" success state on the remaining-balance box (border/check color when `remaining === 0`) |
| Destructive | `var(--destructive)` / `var(--pos-danger)` (red) | Row-remove `Trash2` icon color (`text-destructive`, matches `RecipeEditorTab.tsx`), the amount-mismatch/error banner (existing `border-destructive/40 bg-destructive/10` pattern), and the remaining-balance box when `remaining !== 0` at submit time (overpaid or underpaid) |

Accent reserved for: **primary submit CTA background** and **the "remaining = $0.00" success indicator on the live balance display**. Nowhere else — do not apply `--pos-accent` to per-row method-selector buttons (those keep the existing `variant="default"` → `--primary` / `variant="outline"` scheme, exactly as the single-payment method buttons already do).

## Copywriting Contract

| Element | Copy |
|---------|------|
| Split-mode toggle label | "Split payment" |
| Split-mode toggle helper text | "Divide this total across up to 4 payment methods" |
| Row heading | "Payment {n}" (1-indexed, e.g. "Payment 1", "Payment 2") |
| Add-row CTA | "+ Add payment method" (disabled/hidden once 4 rows exist — mirrors `RecipeEditorTab.tsx`'s "+ Add ingredient") |
| Remove-row control | Icon-only `Trash2`, `aria-label="Remove payment {n}"`; hidden/disabled while only 2 rows remain (split mode requires a minimum of 2 rows — see Open Question A3 in `18-RESEARCH.md`) |
| Live remaining-balance label | "Remaining to pay" while `remaining > 0`; "Fully allocated ✓" when `remaining === 0`; "Over by {amount}" when `remaining < 0` (should not be reachable if per-row validation is correct, but the copy must exist as a safety-net state) |
| Primary CTA (split mode, ready) | "Process split payment" |
| Primary CTA (split mode, in progress) | "Processing…" (reuses existing spinner + copy, unchanged from single-payment flow) |
| Primary CTA (single mode, unchanged) | "Process payment" / "Confirm card payment" / "Confirm & close tab" — **unchanged**, D-11 requires zero regression |
| Empty state (0 rows configured — should not normally occur since split mode starts pre-seeded with 2 rows) | Not applicable — split mode always initializes with 2 default rows on toggle-ON, so a true zero-row empty state is unreachable by design; no empty-state copy needed |
| Error — amount mismatch (client-side, blocks submit) | "Row amounts don't add up to the total yet. Remaining: {formatted amount}." (shown inline near the remaining-balance box, not the generic error banner) |
| Error — empty/zero-amount row (client-side, blocks submit) | "Every payment row needs an amount greater than $0.00." (Pitfall 3 from `18-RESEARCH.md` — client must gate before the DB `amount_positive` CHECK ever fires) |
| Error — backend/RPC failure (generic banner, existing pattern reused) | "Split payment failed — please review each row and try again." (rendered in the existing `role="alert"` destructive banner, same `AlertCircle` + copy pattern as today's single-payment error banner, just with split-specific wording) |
| Receipt sequence header (shown above the reused, unmodified `ReceiptPreview` component) | "Receipt {i} of {n} — {method label}" (e.g. "Receipt 1 of 2 — Efectivo") |
| Destructive actions in this phase | **None.** Removing an unsubmitted split row is not a destructive/data-loss action (matches `RecipeEditorTab.tsx`'s ingredient-row-removal precedent — no confirmation dialog, immediate removal). The split-payment submit itself is gated by the existing `ProtectedAction action="close_tab"` RBAC wrapper and the existing PIN-gate at the `PaymentPane` level (unchanged) — no *additional* confirmation step is introduced by this phase. |

## Interaction Contract (phase-specific, supplements the standard template sections above)

This section documents the row-list mechanics precisely, since it is the primary open design question this spec resolves.

1. **Toggle placement:** `Switch` + label rendered at the top of the existing "Payment method" `<section>` (replaces the method-button grid when ON; the method-button grid returns when OFF). Do not nest the toggle inside a new section — reuse the section's existing `h4`+`space-y-2` wrapper.
2. **Row layout — inline stacked cards (not accordion):** Each row is its own `rounded-lg border p-3` card (exact class reuse from the existing totals-summary box pattern), stacked vertically with `space-y-3` between rows. No collapse/expand interaction — all rows are always fully visible and editable simultaneously, matching `RecipeEditorTab.tsx`'s flat row list.
3. **Row contents, top to bottom:**
   - Row header line: "Payment {n}" (`text-sm font-semibold`) left-aligned, `Trash2` remove button right-aligned (only rendered when `rows.length > 2`).
   - Per-row method selector: 3-button `POSButton` group (`touchSize="large"`, not `xl`), same `variant={selected ? 'default' : 'outline'}` toggle behavior as the single-payment method buttons.
   - Per-row `MoneyInput label="Amount"` (portion of subtotal+tax).
   - Per-row `MoneyInput label="Tip"` (separate, additive — D-06).
   - Method-specific field, conditionally rendered exactly like the single-payment flow does today: cash → `MoneyInput label="Amount tendered"` + change-due readout box; card → `Input` reference-number field (`Label` "Reference # (optional)"); rappi → no extra field (tip section also hidden for rappi rows, mirroring the existing `method !== 'rappi'` gating).
   - Row footer micro-copy (muted, `text-xs`): "Charges: {amount + tip formatted}" — read-only summary of what this specific row will charge the customer.
4. **Add row button:** `POSButton variant="outline"` below the row list, copy "+ Add payment method", disabled when `rows.length >= 4` (D-02 hard cap).
5. **Live remaining-balance box (D-07):** Rendered once, between the row list and the submit button — same visual weight/position as the existing totals-summary box, `rounded-lg border p-3`, label + `MoneyDisplay`. Recomputes on every keystroke in any row's amount field. Accent-green border/check when `remaining === 0`; destructive-red when `!== 0` and the cashier has attempted submit (do not show red before first submit attempt, to avoid alarming the cashier while still typing — matches the existing `canSubmit`-gates-the-button-not-the-display convention already used for cash/card validation in the single-payment flow).
6. **Submit gating:** Primary CTA disabled unless: `2 ≤ rows.length ≤ 4`, every row's `amount > 0`, `Σ(row.amount) === subtotalWithTax` (±0.01 tolerance, matching the RPC's own tolerance per `18-RESEARCH.md`), and every row's method-specific validity holds (cash: `tendered ≥ amount + tip`; card/rappi: no extra requirement beyond `amount > 0`).
7. **Receipts, sequential (D-09):** After a successful split submit, render the existing `ReceiptPreview` component once per leg, unchanged, preceded by a `"Receipt {i} of {n} — {method label}"` header. Clicking the `ReceiptPreview`'s existing "Done" button on any receipt except the last advances `receiptIndex + 1`; on the last receipt it calls the existing `onClose?.()` exactly as the single-payment flow does today. Do not modify `ReceiptPreview` itself — wrap it, don't fork it.
8. **Cash drawer (Pitfall 4 resolution):** Open the physical cash drawer **once per checkout** if `legs.some(l => l.method === 'cash')` — not once per cash leg. `printReceipt` still fires once per leg (D-09 is explicit about that). Flag this as a human-verify checkpoint item during execution per `18-RESEARCH.md`'s Assumption A2 (CONTEXT.md is silent on this specific UX detail).
