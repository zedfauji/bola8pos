# Phase 20 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope Note

This phase adds one new admin surface (Settings → Promotions) and repurposes one existing widget (the happy-hour banner). Per `20-CONTEXT.md` D-06, the admin UI is an explicit clone of the existing Combos admin pattern:

- `src/features/manage-combos/ui/ManageCombosTab.tsx` → clone into `src/features/manage-promotions/ui/ManagePromotionsTab.tsx` (list table + create/delete)
- `src/features/manage-combos/ui/ComboBuilderForm.tsx` → clone into `src/features/manage-promotions/ui/PromotionBuilderForm.tsx` (create/edit dialog fields)
- `src/features/manage-combos/ui/ComboAvailabilityEditor.tsx` → clone **near-verbatim** into `src/features/manage-promotions/ui/PromotionAvailabilityEditor.tsx` (day-of-week + time-window picker; swap `comboId`→`promotionId`, table name only — no visual changes)
- `src/widgets/OrderPanel/HappyHourBanner.tsx` → repurposed in place into a generic "Active Promotions" banner (see Interaction Contract §4 below)

Every visual token below is **inherited** from the Combos admin pattern already in production (verified by direct file read of `ManageCombosTab.tsx`, `ComboBuilderForm.tsx`, `ComboAvailabilityEditor.tsx`, `components.json`, `src/app/globals.css`). Nothing net-new is introduced except two ad-hoc badge colors for the new `target_type` field, extending (not replacing) the existing slot-type badge convention.

**Load-bearing reminder for the executor (from `20-RESEARCH.md` Pitfall 1):** this UI must never let the client compute or submit a discounted price — `evaluate_promotions_for_item` (server) is the sole writer of the charged `unit_price`. Every price/discount value rendered client-side in this phase (list row summaries, the cosmetic banner) is **read-only display**, never fed back into a mutation payload as a final price.

## Typography

Inherited from the Combos admin files — no new type roles introduced.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body / row values (row summary, helper copy, list description) | 14px (`text-sm`) | Regular (400) | 1.5 |
| Label / field names ("Promotion name", "Priority", "Applies to") | 14px (`text-sm`, `font-medium`\*) | Regular (400)\* | 1.5 |
| Section heading (dialog title, "No promotions yet" empty-state heading) | 16px (`text-base`) | Semibold (600, `font-semibold`) | 1.2 |
| Muted / fine print (row sub-text, "No windows = always available", day/time hints) | 12px (`text-xs`, `text-muted-foreground`) | Regular (400) | 1.5 |

\* **Weight resolution note (matches the same tension already resolved in `17-UI-SPEC.md`/`18-UI-SPEC.md`):** `ComboBuilderForm.tsx`'s field labels use `font-medium` (500) and the shared `DialogTitle` primitive (`dialog.tsx:110`) also uses `font-medium` (500) — both are **pre-existing, unmodified shared elements**, not new markup this phase introduces. For any **new** promotions-specific element (the empty-state heading, the target-type badges, any new section subheading), use exactly 2 weights: **Regular (400)** for body/values/muted text, **Semibold (600)** for the one heading role that needs emphasis (mirrors `ManageCombosTab.tsx`'s own `"No combos yet"` empty-state heading, which is `font-semibold text-base` — the real precedent for this phase's empty-state copy). Do not introduce a third weight for new elements; reused shared primitives (labels, `DialogTitle`) keep their existing 500-weight unchanged.

## Color

Inherited from `src/app/globals.css` (oklch CSS variables, dark mode default). No new colors introduced.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `var(--background)` (`oklch(0.145 0 0)` dark) | Settings page background, dialog background |
| Secondary (30%) | `var(--card)` / `var(--muted)` via `rounded-md border` boxes | List row divider box (`divide-y rounded-md border`), each availability-window card, the create-form's bordered sections |
| Accent (10%) | `var(--primary)` | Reserved for exactly: (1) the **"+ Add promotion"** button, (2) the **"Save promotion"** dialog submit button, (3) the day-of-week toggle **selected** state in the reused availability editor (`bg-primary text-primary-foreground`, verbatim from `ComboAvailabilityEditor.tsx`'s `WindowRow`), (4) the **Active** `Switch` "on" track color (shadcn default). Not used for row backgrounds, borders, or the Edit/Delete outline buttons (those stay `variant="outline"`, unchanged from the Combos precedent). |
| Destructive | `var(--destructive)` | Row-level `Trash2` delete icon (`text-destructive`), the "Delete Promotion" `ConfirmDialog`, the availability editor's "End time must be after start time" validation text (verbatim reuse, unchanged) |

Accent reserved for: **the "+ Add promotion" button, the "Save promotion" button, the selected day-toggle chips inside the availability editor, and the Active-toggle Switch.** Nowhere else.

**New ad-hoc badge colors (target-type only)** — extends the existing convention already live in `ComboBuilderForm.tsx`'s `slotTypeBadge` (`bg-blue-500/20 text-blue-300` for `'product'`, `bg-purple-500/20 text-purple-300` for `'pool_time'`). This phase's `target_type` has 4 values instead of combo's 2, so 2 more colors from the same low-saturation `/20`-opacity family are added:

| `target_type` | Badge classes |
|----------------|---------------|
| `item` | `bg-blue-500/20 text-blue-300` (reuses combo's `'product'` color exactly) |
| `category` | `bg-purple-500/20 text-purple-300` (reuses combo's `'pool_time'` color — acceptable since combo and promotion badges never render side by side) |
| `pool_billing` | `bg-cyan-500/20 text-cyan-300` (new) |
| `pool_grant` | `bg-amber-500/20 text-amber-300` (new) |

These 4 colors are decorative categorization only — never used to convey pricing/discount magnitude, and never rendered anywhere except the promotions list row and the builder form's live preview of the selected target type.

## Interaction Contract (phase-specific, supplements the standard template sections above)

### 1. `ManagePromotionsTab` (list) — clone of `ManageCombosTab.tsx`
- Header row: description text (left) + "+ Add promotion" button (right), `flex items-center justify-between gap-2` — verbatim layout.
- Empty state: bordered box (`rounded-md border px-4 py-10 text-center space-y-2`), heading + body + CTA — verbatim layout.
- Populated list: `<ul className="divide-y rounded-md border">`, each `<li className="flex items-center gap-3 px-3 py-2.5">` containing: name (truncate) + row summary sub-text (left, flex-1), target-type badge (new), Active `Switch` (new, inline), Edit button, Delete button (right cluster, `flex items-center gap-1`).
- Clicking "+ Add promotion" creates a draft row (`is_active: false` default, matching the "safe to leave incomplete" pattern combo uses for `is_combo: true` drafts) and immediately opens the edit dialog — verbatim `handleCreate` pattern.

### 2. `PromotionBuilderForm` (create/edit dialog content) — clone of `ComboBuilderForm.tsx`
Field order, top to bottom (mirrors the combo form's name→price→slots ordering, substituting promotion-specific fields):
1. Promotion name (`Input`, required, `maxLength={100}`)
2. Discount type (`Select`, 3 options — see Copywriting Contract)
3. Discount value (`MoneyInput` for `fixed_amount`/`fixed_price`; plain `Input type="number" min={0} max={100} step={1}` with a `%` suffix hint for `percentage`) — conditionally rendered based on step 2's selection, exactly like `ComboBuilderForm.tsx`'s slot-type-conditional option picker
4. Fixed-price stacking hint (conditional, `discount_type === 'fixed_price'` only — see Copywriting Contract)
5. Target type (`Select`, 4 options — see Copywriting Contract)
6. Target picker (conditional: native `<select>` for `item`, `CategoryTreePicker` for `category`, hidden for `pool_billing`/`pool_grant`)
7. Priority (`Input type="number" min={0} step={1}`, default `0`)
8. Active (`Switch` + "Active" label, default `off` for new drafts)
9. Save button, right-aligned (`flex justify-end`) — verbatim position

Below the form, inside the same dialog (mirrors `ManageCombosTab.tsx`'s edit-dialog composition of `ComboBuilderForm` + a `border-t pt-4` divider + `ComboAvailabilityEditor`):
10. `border-t pt-4` divider
11. `PromotionAvailabilityEditor` — the reused day/time picker (see §3)

### 3. `PromotionAvailabilityEditor` — clone of `ComboAvailabilityEditor.tsx`, near-verbatim
No visual changes. Swap `comboId`→`promotionId` and the target table name only. Day toggle chips, time inputs, "No windows = always available" messaging (top and bottom), "+ Add window" / "Save" buttons — all pixel-for-pixel identical to the combo version. This is the literal reuse `20-CONTEXT.md` D-06 asks for.

### 4. "Active Promotions" banner — repurposed `HappyHourBanner.tsx`
Per D-01, the existing `HappyHourBanner.tsx` is generalized rather than rewritten from scratch, preserving its exact visual shape:
- Same container: `rounded-lg border border-amber-700 bg-amber-950 px-3 py-2 text-sm text-amber-200`, `role="status"`.
- Same icon: `Zap` (amber), same live-tick-per-second countdown mechanism (`setInterval` 1000ms).
- **Copy change:** `"Happy Hour Active — {names}"` → `"Promotions Active — {names}"`, where `{names}` lists the currently-active promotion names (comma-joined), not category names.
- **Countdown behavior:** unchanged when the active promotion(s) have a known `end_time` for the current window (`"Ends in {countdown}"`); when a matching promotion has **no** time window at all (`combo_availability`-style "no windows = always available" per Pattern 1 in `20-RESEARCH.md`), omit the "Ends in…" suffix entirely rather than showing a misleading `Infinity`/`0m` — this is a real gap in the current `HappyHourBanner.tsx` logic (`endMinutes` defaults to `Infinity` per-category with no explicit "always on" copy path) that must be fixed as part of the repurposing, not carried forward.
- **Prop shape change:** `categories: Category[]` → a new prop shape sourced from active promotions (exact hook/type left to the planner's entity-layer design — this spec only locks the visual/copy contract).
- **Test id:** `data-testid="happy-hour-banner"` → `data-testid="active-promotions-banner"`. Flag to the planner: this rename requires updating any existing test selectors that reference the old testid.
- **Cosmetic-only guarantee (Pitfall 1):** this banner's active/inactive determination and any displayed discount amount are read-only display, driven by a client-side `isPromotionActive()` helper (mirrors the retiring `isHappyHourActive()`) — never the source of truth for the charged price, and never re-used to compute a value sent to `create_order_with_items`.

## States Checklist (for executor)

**`ManagePromotionsTab`:**
- [ ] Loading: `text-sm text-muted-foreground` "Loading promotions…" (verbatim pattern)
- [ ] Error: `text-sm text-destructive` "Could not load promotions: {message}"
- [ ] Empty (0 promotions): bordered empty-state box, heading + body + CTA
- [ ] Populated (N promotions): row list, each row = name + summary + target-type badge + Active switch + Edit + Delete
- [ ] Create-in-flight: "+ Add promotion" button shows "Creating…" (mirrors "Creating…" on Combos' create button), disabled while pending
- [ ] Delete confirm open → `ConfirmDialog` destructive variant, `isLoading` while pending

**`PromotionBuilderForm`:**
- [ ] Loading (fetching existing promotion for edit): "Loading promotion…" (verbatim pattern)
- [ ] Discount-value field swaps input type/component when discount type changes (percentage ↔ MoneyInput fields) — no leftover stale value from the previous type; clear on type switch
- [ ] Target picker swaps (native select ↔ CategoryTreePicker ↔ hidden) when target type changes
- [ ] Fixed-price hint appears/disappears immediately on discount-type change (no stale hint lingering after switching away from `fixed_price`)
- [ ] Saving: Save button shows "Saving…", disabled
- [ ] Save success: `toast.success('Promotion saved')`
- [ ] Save failure: `toast.error(...)`, dialog stays open, no data loss

**`PromotionAvailabilityEditor`:** identical to `ComboAvailabilityEditor.tsx`'s existing states (loading / 0 windows / N windows / time-range validation error / saving / save success / save failure) — no new states, verbatim reuse.

**"Active Promotions" banner:**
- [ ] No active promotions: renders `null` (verbatim `HappyHourBanner.tsx` early-return behavior)
- [ ] 1+ active promotions, all with a known end time: shows "Ends in {countdown}" using the earliest end time (verbatim logic, renamed copy)
- [ ] 1+ active promotions, at least one with **no** time window ("always available"): omits the "Ends in…" suffix entirely (bug-fix behavior locked by this spec — see Interaction Contract §4)
- [ ] Live countdown ticks every second without a full component remount (verbatim `setInterval` mechanism)
