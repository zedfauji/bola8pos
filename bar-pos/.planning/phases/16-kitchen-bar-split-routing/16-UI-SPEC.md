# Phase 16 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Phase type: extension, not greenfield.** Per D-06/D-07 this phase reuses `KdsBoard`/`KdsCard`/`useKdsItems` verbatim, parameterized by a routing filter — it does NOT rebuild the KDS board. The only genuinely new visual surfaces are: (1) `RoutingBadge` (shared/ui) and (2) a routing selector added to `CategoryForm` inside `CategoryTreeEditor.tsx` (this field does not exist today — `isFood` is currently hardcoded `false` on create and has no edit control at all). This contract's job is: exact `RoutingBadge` visual spec, exact placement/copy for the new routing selector, and the minimal-diff copy changes `KdsBoard`/`KdsCard` need to stay generic across `/kds` and `/kds-bar`.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `bg-background` / `text-foreground` | Page background for `/kds-bar` (identical to `/kds`), `CategoryTreeEditor` panel background |
| Secondary (30%) | `bg-card` / `bg-muted` / `bg-secondary` | `RoutingBadge` background (both KITCHEN and BAR states — see below), `Select` trigger background (`bg-background` with `border-input`, per existing `select.tsx`) |
| Accent (10%) | `bg-primary` / `text-primary-foreground` | **Reserved for:** none newly introduced this phase — no primary/accent-styled button is added. The existing bump `Button` (`variant="secondary"`/`variant="default"`) in `KdsCard` is reused unchanged. |
| Destructive | `bg-destructive` / `text-destructive` | Reserved for: the existing "Could not load {queue}" error copy + Retry button pattern in `KdsBoard`, reused unchanged for `/kds-bar` |

Accent reserved for: nothing new. This phase deliberately does **not** spend the 10% accent budget on `RoutingBadge` — the card's color budget is already committed to (a) the pending/in-progress/done status border (`yellow-500`/`blue-500`/`green-500`, existing) and (b) `ComboBadge`'s `pos-accent` green tint (existing). Introducing a third color axis on the same card (e.g. green-for-BAR vs orange-for-KITCHEN) would create ambiguity with the existing status border color and with `ComboBadge`'s green. `RoutingBadge` MUST differentiate KITCHEN vs BAR by **icon + text only**, using the same neutral `variant="secondary"` treatment for both states (identical to how `ChefHatBadge` uses neutral gray + icon for "Prep" rather than inventing a new hue).

`RoutingBadge` color/icon states:

| Routing value | Badge shown? | Icon | Text | Variant |
|----------------|--------------|------|------|---------|
| `KITCHEN` | Yes | `UtensilsCrossed` (lucide) — same icon already used for the `/kds` HomeDashboard tile, reused for semantic consistency | "Kitchen" | `secondary` (neutral, matches `ChefHatBadge`) |
| `BAR` | Yes | `Beer` (lucide) | "Bar" | `secondary` (neutral, matches `ChefHatBadge`) |
| `NONE` | No badge on `KdsCard` (moot — `NONE` items never appear in either KDS query per Claude's-discretion default). In `CategoryTreeEditor`'s `NodeRow`, render plain muted text **"Not routed"** (`text-xs text-muted-foreground`) instead of a badge — this keeps "intentionally unrouted" visually distinct from "badge missing due to a bug" | — | "Not routed" (plain text, not a `Badge`) | n/a |

| Destructive actions in this phase | **None.** Changing a category's routing is a config edit equivalent to editing its name/color (same `CategoryForm` submit flow, no confirmation step) — consistent with the existing form, which has no confirm step for name/color edits either. **Caution for planner/executor (not a UI requirement, a functional flag):** switching a category from `KITCHEN`/`BAR` to `NONE` (or between `KITCHEN`↔`BAR`) will cause any already-pending order items in that category to silently disappear from both KDS boards on next query refresh, since the routing filter is applied live at query time rather than snapshotted per order-item. ## Screen-by-Screen Contract

### A. `RoutingBadge` (new, `src/shared/ui/RoutingBadge.tsx`)

Model directly on `ChefHatBadge.tsx` (icon + `Badge variant="secondary"`), not `ComboBadge` (which uses the reserved accent-green tint — deliberately not spent here, see Color section above):

```tsx
import { Beer, UtensilsCrossed } from 'lucide-react';
import { Badge } from './badge';
import type { CategoryRouting } from '@shared/lib/domain'; // 'KITCHEN' | 'BAR' | 'NONE'

export function RoutingBadge({ routing }: { routing: CategoryRouting }) {
  if (routing === 'NONE') return null;
  const Icon = routing === 'KITCHEN' ? UtensilsCrossed : Beer;
  const label = routing === 'KITCHEN' ? 'Kitchen' : 'Bar';
  return (
    <Badge variant="secondary" className="flex items-center gap-1">
      <Icon className="h-3 w-3" aria-hidden />
      {label}
    </Badge>
  );
}
```

Returns `null` for `NONE` — callers needing the "Not routed" text state (i.e. `CategoryTreeEditor`'s `NodeRow`) render that plain-text fallback themselves rather than `RoutingBadge` special-casing a text mode; keeps the badge component single-purpose (a badge is either shown or it isn't).

### B. `KdsCard` / `ComboKdsCard` — add `RoutingBadge` next to product name

Both card variants in `src/widgets/KdsBoard/index.tsx` currently render the product name alone (`KdsCard`) or product name + `ComboBadge` (`ComboKdsCard`). Add `RoutingBadge` immediately after the name, before `ComboBadge` where both are present:

```tsx
<div className="flex items-center gap-2">
  <p className="truncate text-lg font-bold">{item.productName}</p>
  <RoutingBadge routing={item.routing} />
  {/* ComboKdsCard only: */}
  <ComboBadge />
</div>
```

This requires `KdsOrderItem` (in `entities/kds/model/types.ts`) to carry a `routing` field (currently only `isFood: boolean`) — a data-shape change the planner must sequence alongside D-01/D-07's schema migration, not a new decision this UI-SPEC introduces.

Per D-08's own framing ("useful once/if any cross-listing exists"), every card on a single board render will show the same routing value today (all "Kitchen" on `/kds`, all "Bar" on `/kds-bar`) — this is expected and correct, not a bug; the badge exists for forward-compatibility, not because it's currently informative on every card.

### C. `/kds-bar` page (new, `src/pages/kds-bar/index.tsx`)

Copy `src/pages/kds/index.tsx` exactly, changing only the title and the `KdsBoard` routing prop:

```tsx
import { KdsBoard } from '@widgets/KdsBoard';
import { BackToHomeButton, LiveTimeDisplay, PageContainer } from '@shared/ui';

export default function KdsBarPage() {
  return (
    <div className="flex h-screen flex-col">
      <BackToHomeButton />
      <main className="flex-1 overflow-auto">
        <PageContainer title="Bar Display" actions={<LiveTimeDisplay />}>
          <KdsBoard routing="BAR" />
        </PageContainer>
      </main>
    </div>
  );
}
```

`KdsBoard` gains a `routing: 'KITCHEN' | 'BAR'` prop (threaded to `useKdsItems(routing)` per D-06/D-07); `/kds/index.tsx` is updated to pass `routing="KITCHEN"` explicitly (no default — explicit at both call sites avoids an implicit-default footgun when a third routing value or cross-listing mode is added later).

### D. `KdsBoard` — parameterized empty/error copy

`KdsBoard`'s hardcoded strings become derived from the `routing` prop:

```tsx
const stationLabel = routing === 'KITCHEN' ? 'kitchen' : 'bar';
// Empty: `No active ${stationLabel} orders`
// Error:  `Could not load ${stationLabel} queue.`
```

No other visual change to `KdsBoard`'s layout, grid, or column structure — the two-column Pending/In Progress grid, section badges (yellow pending count / blue in-progress count), and bump-button styling are all reused byte-for-byte across both stations.

### E. `KdsBarRoute` guard (new, `src/app/kds-bar-route.tsx`)

Copy `src/app/kds-route.tsx` exactly, swapping the permission check per D-04/D-05 (no visual component — purely a guard, no dialog/toast on redirect, matching `KdsRoute`'s silent-redirect precedent, NOT `AuditRoute`'s toast-on-redirect variant):

```tsx
import type { ReactNode } from 'react';
import { Navigate } from 'react-router-dom';
import { usePermissions } from '@entities/staff/model/usePermissions';

export function KdsBarRoute({ children }: { children: ReactNode }) {
  const { can } = usePermissions();
  if (!can('view_kds_bar')) {
    return <Navigate to="/home" replace />;
  }
  return <>{children}</>;
}
```

### F. HomeDashboard — new `/kds-bar` tile

Add alongside the existing `/kds` tile in `ITEMS` (`src/widgets/HomeDashboard/ui/HomeDashboard.tsx`). Unlike the pre-existing `/kds` tile (which uses the older `visibleToRoles: ['admin', 'kitchen']` pattern), the new tile MUST use the `requiredAction` pattern already standard for every other manager+-gated tile (Reports, Inventory, Settings, Kitchen Prep, Waitlist, Roles & Permissions, Audit Log) — this is the correct/current convention, not a new one:

```tsx
{
  path: '/kds-bar',
  label: 'Bar Display',
  icon: Beer,
  requiredAction: 'view_kds_bar',
  managerLabel: 'Bartender',
},
```

`managerLabel: 'Bartender'` is a new value for this field (existing values are 'Manager'/'Admin' only) — acceptable, the field is a free-form badge label rendered next to gated tiles; confirm the badge component that renders `managerLabel` doesn't hardcode an enum (verify at implementation time; if it does, extend it rather than misusing 'Manager').

### G. `CategoryForm` — new "Routing station" selector (`CategoryTreeEditor.tsx`)

Insert a new field between "Color" and the Cancel/Save button row, following the exact `FormField` wrapper pattern already used for Name/Color:

```tsx
<FormField label="Routing station">
  <Select value={routing} onValueChange={setRouting}>
    <SelectTrigger id="cat-routing">
      <SelectValue placeholder="Select routing" />
    </SelectTrigger>
    <SelectContent>
      <SelectItem value="KITCHEN">
        <UtensilsCrossed className="h-3.5 w-3.5" aria-hidden /> Kitchen
      </SelectItem>
      <SelectItem value="BAR">
        <Beer className="h-3.5 w-3.5" aria-hidden /> Bar
      </SelectItem>
      <SelectItem value="NONE">None</SelectItem>
    </SelectContent>
  </Select>
  <p className="mt-1 text-xs text-muted-foreground">
    Controls which KDS board this category's items appear on. Choose &ldquo;None&rdquo; for items
    that aren&rsquo;t prepped (e.g. merch).
  </p>
</FormField>
```

`CategoryFormData` gains a `routing: CategoryRouting` field (default `'NONE'` for new root/child categories — a safe default that excludes new categories from both boards until an admin deliberately opts them in, rather than defaulting to `'KITCHEN'` and silently spamming the kitchen board with unrouted junk categories). `handleCreate`'s hardcoded `isFood: false` is replaced with `routing: data.routing`. `handleUpdate` — which currently does NOT persist `isFood` at all (a pre-existing gap, since the field is absent from the update payload) — must now include `routing` in its update payload; this is the first time routing/isFood becomes editable post-creation, closing a real gap, not just adding an enum.

### H. `NodeRow` — `RoutingBadge` / "Not routed" placement

Insert immediately after the category name span, before the existing depth badge (`L{depth+1}`):

```tsx
<span className="flex-1 text-sm font-medium">{category.name}</span>
{category.routing === 'NONE' ? (
  <span className="text-xs text-muted-foreground">Not routed</span>
) : (
  <RoutingBadge routing={category.routing} />
)}
<span className="text-xs text-muted-foreground">L{depth + 1}</span>
```

No change to the row's overall height/spacing rhythm (`gap-2`, `py-1.5` preserved) — the badge/text fits inside the existing row without introducing a second line.
