# Phase 7 — UI Design Contract: Waitlist + WhatsApp

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

> **MANDATORY AGENT GUARDRAIL — E2E runs and browser console**
>
> On **every** E2E test run (Playwright, CI, or agent retry loops), **tail or otherwise capture the browser console for that run** and **read it before concluding why a test failed or before re-running the same spec**. Failures are often explained only in the console (uncaught exceptions, failed network calls, React errors, hydration warnings). **Do not** repeatedly execute the same failing E2E in a tight loop without console evidence — that burns tokens and time while the real signal sits in logs the agent never opened. Treat "console captured and reviewed for this run" as **non-optional** and part of the same step as "test run completed."

### HomeDashboard Waitlist Tile

Add to ITEMS array in `HomeDashboard.tsx`. Tile renders a dynamic waiting-count badge overlay.

```typescript
// In HomeDashboard.tsx — item definition (static config):
{
  path: '/waitlist',
  label: 'Waitlist',
  icon: ListOrdered,   // or Clock3 if ListOrdered unavailable
  requiredAction: 'manage_waitlist',
  managerLabel: 'Manager',
}
```

The waiting-count badge is a numeric overlay on the tile icon. Since `HomeDashboard` needs a live count, the component must call `useWaitlistWaitingCount()` (a lightweight query returning the count of `status='waiting'` entries). The badge renders only when `count > 0`:

```tsx
// Inside the tile button, overlaid on the icon:
{waitingCount > 0 && (
  <Badge
    variant="destructive"
    className="absolute -right-1 -top-1 h-5 min-w-[20px] rounded-full px-1 text-sm"
  >
    {waitingCount > 99 ? '99+' : waitingCount}
  </Badge>
)}
```

The `Badge` is positioned `absolute` relative to the `Icon` container (which needs `relative` class). This is consistent with the `Lock` icon overlay already in `HomeDashboard`.

## Interaction Contracts

### AddWaitlistEntryForm — state machine

```
CLOSED
  → ["Add to waitlist" button tap] → OPEN (all fields empty; party size defaults to 1)

OPEN
  Field states:
    name: empty (invalid) | filled (valid when trimmed length >= 1)
    partySize: always valid (Select with default "1")
    phoneRaw: empty (valid — optional) | filled+valid | filled+invalid

  isValid = name.trim().length >= 1 && (phoneRaw === '' || toE164(phoneRaw) !== null)

  Phone validation: fires on blur only; error shown inline; cleared immediately on edit

  SUBMITTING (isValid = true, "Add to waitlist" tapped)
    → Both buttons disabled; "Add to waitlist" shows LoadingSpinner
    → [Success] → CLOSED + "{name} added to the waitlist." toast
    → [Error] → OPEN + error toast; form state preserved; user can correct and retry
```

### NotifyButton — state machine

```
DEFAULT
  → [Notify button tap] → PENDING
    entry status update to 'notified' sent to Supabase

PENDING
  → Button disabled; LoadingSpinner visible
  → [Success] → toast shown; Realtime update causes WaitlistQueue to re-render card with 'notified' status; button no longer shown
  → [Error] → button re-enabled; error toast shown
```

### SeatPartySheet — state machine

```
CLOSED
  → ["Seat party" button tap in WaitlistEntryCard] → OPEN (no table selected)

OPEN
  → [Table card tap] → table selected (highlighted border-pos-accent bg-pos-accent/10)
  → [Different table tap] → previous selection cleared; new table selected
  → ["Seat party" button tap (table selected)] → SUBMITTING

SUBMITTING
  → Both buttons disabled; "Seat party" shows LoadingSpinner
  → [Success] → CLOSED + "{name} seated at {tableName}." toast; Realtime update refreshes queue
  → [Error] → OPEN + error toast; selected table preserved
```

### No-show confirmation — flow

```
[UserX icon tap in WaitlistEntryCard]
  → ConfirmDialog opens (AlertDialog)
  → [Cancel / overlay dismiss] → dialog closed; no action
  → [Confirm "Mark no-show"] → dialog closed; useMarkNoShow mutation fires
    → [Success] → "{name} marked as no-show." toast; card removed from active queue
    → [Error] → error toast; card remains
```

```
  → ConfirmDialog opens
  → [Cancel] → dialog closed; no action
  → [Confirm "Cancel entry"] → useMarkCancelled mutation fires
    → [Success] → "Entry for {name} cancelled." toast; card removed
    → [Error] → error toast; card remains
```

### WaitlistPage — real-time behavior

- `WaitlistRealtimeListener` (app-level component) subscribes to `waitlist_entries` and `pool_tables` Postgres changes via one Supabase Realtime channel (`waitlist:pos-sync`).
- On any change: `invalidateQueries({ queryKey: waitlistKeys.all })` and `invalidateQueries({ queryKey: poolTableKeys.all })`.
- On broadcast `notified` event from edge function: same invalidation.
- Queue re-renders automatically; `quotedWait` recomputes per entry from updated snapshot.
- No manual refresh button. No optimistic update for card reordering — let Realtime drive truth.

### Quoted wait — display rules

- Show "~{N} min wait" computed from `computeQuotedWait()`.
- If `quotedWait` returns `<= 5` (minimum floor), show "~5 min wait".
- If `quotedWait > 120`, show ">2 hr wait" (cap display at 2h for UX clarity).
- Never show `0 min wait` — the floor is 5.

### RBAC gate

- `/waitlist` route guarded by `WaitlistRoute` (mirrors `KitchenPrepRoute` pattern exactly).
- Action required: `manage_waitlist` (manager+ only).
- If `can('manage_waitlist')` is false: redirect to `/home`.
- Home tile: rendered for all roles; shows `Lock` icon when access is gated (inherits existing `HomeDashboard` pattern). Tapping triggers `ManagerPinDialog`.
