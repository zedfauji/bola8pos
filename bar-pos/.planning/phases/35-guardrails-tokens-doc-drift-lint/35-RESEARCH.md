## Summary

This phase has two deliverables — `DESIGN-TOKENS.md` (doc) and `eslint-rules/no-ui-drift.js` + `eslint-plugin-tailwindcss` (lint) — both mechanically simple. The real risk is **scope creep in the lint rule**: the phase's premise (D-09, D-14) is "the codebase already conforms, so `error` severity is safe." That premise is only true for the narrow slice Phase 29's audit script actually scanned (`src/pages`, `src/widgets`, `src/features`, excluding `src/shared/ui`) and only for the 4 categories it scanned with its specific regexes. Live re-verification in this session found it is **not fully true even in that scope**: a `manage-promotions` feature (not in CLAUDE.md, not covered by Phases 30-33) has an unconverted raw `<button>` day-picker, and `eslint-plugin-tailwindcss`'s `no-arbitrary-value` rule flags **any** bracketed Tailwind value (not just spacing) — meaning turning it on verbatim would newly flag ~70 pre-existing `h-[...]`/`w-[...]`/`min-h-[...]` arbitrary-size classes across ~40 files that Phase 29's narrower spacing-only regex never touched and Phases 30-33 never fixed.

Three additional non-obvious ESLint mechanics matter for how the plan should be structured: (1) flat-config `rules` keys **replace**, not merge, across config objects for the same matched files — if `eslint-rules/no-ui-drift.js`'s selectors are added as a *second* `no-restricted-syntax` config object matching the same files as the existing barrel-export ban, the existing ban silently disappears; (2) `eslint-plugin-tailwindcss`'s `callees` option (which functions to scan string arguments inside) defaults to `['classnames', 'clsx', 'ctl', 'cva', 'tv']` and does **not** include this project's `cn()` helper — without adding it explicitly, ~11 files whose classes are built via `cn(...)` are invisible to the plugin's rules; (3) Phase 31 already pre-negotiated and partially pre-commented three categories of *intentional, permanent* exceptions (native `type="color"/"time"/"date"/"file"` inputs, signed-delta number inputs, and `category.color` hex user-data) that the new rule must not re-flag — the existing comments are prose only, not `eslint-disable` directives, so they will not actually suppress anything until converted.

**Primary recommendation:** Scope every new rule (both the `eslint-plugin-tailwindcss` block and `eslint-rules/no-ui-drift.js`) to `files: ['src/pages/**/*.tsx', 'src/widgets/**/*.tsx', 'src/features/**/*.tsx']` — exactly Phase 29's `SCAN_ROOTS` — so `src/entities/**` and `src/shared/ui/**` (both known to carry pre-existing, never-audited drift) stay out of scope by construction, matching D-09/D-12's actual historical boundary rather than the broader "everywhere except shared/ui" reading. Do **not** rely on `no-arbitrary-value`'s bare defaults; treat its blanket bracket-matching as a hard constraint that requires either (a) an explicit inline `eslint-disable-next-line` at each of the ~4 known touch-target/legitimate sites plus remediation of the ~70 non-touch-target arbitrary size classes (large, out of this phase's "no visual changes" mandate), or (b) implementing the spacing-drift check as a narrower custom `no-restricted-syntax` regex mirroring Phase 29's exact prefix list (`p|px|py|pt|pb|pl|pr|m|mx|my|mt|mb|ml|mr|gap|gap-x|gap-y|space-x|space-y`) instead of enabling `no-arbitrary-value` — this is a plan-level decision the planner must make explicitly (see Open Questions).

### Pattern 1: Flat-config `no-restricted-syntax` selectors must be merged into ONE array, not declared in a second competing config object

**What:** ESLint flat config resolves the final rule set for a file by shallow-merging all matching config objects **in array order**; for a single rule key like `no-restricted-syntax`, the *last* config object that sets it for a given file **completely replaces** the earlier value — arrays of selectors do not concatenate across config objects.

**Why this matters here:** `eslint.config.js` already has:
```js
'no-restricted-syntax': [
  'error',
  { selector: 'ExportAllDeclaration', message: 'Barrel exports (export *) are banned...' },
],
```
on the block matching `files: ['**/*.{ts,tsx}']`. If `eslint-rules/no-ui-drift.js` is wired in as a **second** config object that also sets `no-restricted-syntax` for files under `src/pages|widgets|features` (which are also matched by the first, broader block), the second block's value **silently replaces** the first — the barrel-export ban would stop firing for every file in `pages/widgets/features` with no error, no warning, just a rule that quietly stops working.

**How to avoid:** Export an array of selector objects from `eslint-rules/no-ui-drift.js` (not a full rule config), and spread it into the *same* `no-restricted-syntax` array already in `eslint.config.js`:
```js
// eslint-rules/no-ui-drift.js
export const uiDriftSelectors = [
  { selector: "JSXOpeningElement[name.name='button']", message: '...' },
  { selector: "JSXOpeningElement[name.name='input']:not(:has(JSXAttribute[name.name='type'][value.value=/^(color|time|date|file)$/]))", message: '...' },
  { selector: 'Literal[value=/^#([0-9a-fA-F]{3}){1,2}$/]', message: '...' },
];
```
```js
// eslint.config.js
import { uiDriftSelectors } from './eslint-rules/no-ui-drift.js';
// ...
'no-restricted-syntax': [
  'error',
  { selector: 'ExportAllDeclaration', message: '...' },
  ...uiDriftSelectors,
],
```
If a *scoped-only* application (pages/widgets/features, not the whole repo) is required, add a **second, narrower config object** but restate `ExportAllDeclaration` in it too — or better, keep the barrel-export selector applying repo-wide (as now) and add the UI-drift selectors as extra entries in the SAME array, since `no-restricted-syntax` selectors that don't match a given file's AST simply never fire (a `JSXOpeningElement` selector is a no-op in a `.ts` file with no JSX) — no separate `files` scoping is strictly required for the custom selectors, only for the `eslint-plugin-tailwindcss` plugin block (see Pattern 2).

### Pattern 2: Scope `eslint-plugin-tailwindcss` rules to a `files` glob matching Phase 29's audited zone

**What:** Add the plugin as a flat-config object with `files: ['src/pages/**/*.tsx', 'src/widgets/**/*.tsx', 'src/features/**/*.tsx']`, not the repo-wide default.

**When to use:** Always, for this phase — `src/entities/**` and `src/shared/ui/**` are known (see Common Pitfalls) to carry pre-existing arbitrary-value and custom-classname usage that predates and falls outside Phase 29's audit scope; including them will break the "codebase already conforms" premise the phase depends on.

**Example:**
```js
// Source: https://github.com/francoismassart/eslint-plugin-tailwindcss (v3.18.3 README, flat-config section)
import tailwind from 'eslint-plugin-tailwindcss';

export default tseslint.config(
  // ...existing config objects...
  {
    files: ['src/pages/**/*.tsx', 'src/widgets/**/*.tsx', 'src/features/**/*.tsx'],
    plugins: { tailwindcss: tailwind },
    settings: {
      tailwindcss: {
        config: './tailwind.config.ts',
        callees: ['cn', 'clsx', 'classnames', 'ctl', 'cva', 'tv'], // 'cn' is this repo's helper — NOT in plugin defaults
        whitelist: ['^(animate|fade|slide|zoom)-(in|out)(-from-\\w+)?$'], // tw-animate-css utility classes
      },
    },
    rules: {
      'tailwindcss/no-arbitrary-value': 'error',
      'tailwindcss/no-custom-classname': 'error',
      'tailwindcss/enforces-shorthand': 'error',
    },
  },
);
```

### Anti-Patterns to Avoid
- **Extending `eslint-plugin-tailwindcss`'s `configs['flat/recommended']` wholesale:** pulls in `classnames-order` and other rules D-08 didn't ask for, and applies repo-wide by default — declare the three named rules explicitly instead, scoped per Pattern 2.
- **Treating `no-arbitrary-value`'s absence of a per-value allowlist as "add an eslint-disable per site and move on":** with ~70 non-touch-target arbitrary size classes in the audited zone (see Pitfall 1), this is not a small number of disables — it is a scope decision that needs to be made explicitly, not discovered mid-execution.
- **Writing the 3 custom `no-restricted-syntax` selectors as prose-only regex reinvention:** reuse Phase 29's exact category patterns (`scripts/audit-ui-drift.ts` lines 26-40) for the button/input/hex categories where they map 1:1 to a `no-restricted-syntax` selector; only the arbitrary-spacing category needs a decision (plugin rule vs. custom regex — see Open Questions), not the other three.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Tailwind arbitrary-value / custom-classname detection | A regex scanner reinventing Tailwind's class-name grammar | `eslint-plugin-tailwindcss`'s `no-arbitrary-value`/`no-custom-classname` (AST-aware, understands Tailwind's config-driven valid-class set) | Regex can't distinguish a valid dynamic `data-[state=open]:` variant from a truly custom/typo'd class; the plugin parses against the real `tailwind.config.ts` |
| Reading `tailwind.config.ts`'s token values | Regex-scraping the `.ts` file as text | Dynamic `import()` of the config module at generator-script runtime (the script already runs via `tsx`, which resolves `.ts` imports natively) | `tailwind.config.ts` is a real TS module exporting a plain object — importing it gives structured, always-in-sync access to `theme.extend.colors`/`borderRadius`; regex-scraping (as `audit-ui-drift.ts` does for a different, presence/absence purpose) would silently drift from the real values on any config refactor |

**Key insight:** Both plugin (for Tailwind-class semantics) and native `import()` (for reading a real TS config module) already solve exactly this phase's two mechanical problems — no scanner needs to be invented from scratch. The one place hand-rolling is correct (D-10) is the 3 project-specific `no-restricted-syntax` selectors (raw button/input/hex), because no generic plugin encodes "this project's `POSButton`/`shared/ui` convention."

## Common Pitfalls

### Pitfall 1: `no-arbitrary-value` flags every bracketed Tailwind value, not just spacing — the codebase is not conformant on this once `min-h`/`min-w`/`h`/`w` are included
**What goes wrong:** Phase 29's `audit-ui-drift.ts` scanner (the source of D-09's "already conformant" claim) only regex-matched spacing prefixes: `p|px|py|pt|pb|pl|pr|m|mx|my|mt|mb|ml|mr|gap|gap-x|gap-y|space-x|space-y` (verified: `scripts/audit-ui-drift.ts` lines 34-39). It found 0 files. `eslint-plugin-tailwindcss`'s `no-arbitrary-value` rule has no such restriction — its matching logic is `/\[.*\]/i.test(parsed.body)` [VERIFIED: source `lib/rules/no-arbitrary-value.js`, v3.18.3] — it flags **any** utility with bracket syntax: `min-h-[...]`, `min-w-[...]`, `h-[...]`, `w-[...]`, `rounded-[...]`, `text-[...]`, etc.
**Why it happens:** The plugin was never designed to match Phase 29's narrower "spacing drift" definition — it's a general-purpose arbitrary-value linter.
**How to avoid:** See Open Questions #1 — this is a plan-level scope decision, not a code fix.
**Warning signs:** Live grep in this session found **70 occurrences** of `h-`/`w-`/`min-h-`/`min-w-`/`max-h-`/`max-w-` with bracket values outside the 3 canonical touch-target values (`min-h-[44px]`/`[56px]`/`[72px]`), across ~40 files in `src/pages|widgets|features|entities|shared` — e.g. `min-h-[12rem]`/`[14rem]` (Settings tab wrappers), `min-h-[80px]`/`[120px]`/`[160px]`/`[200px]` (card/empty-state min-heights), `h-9 w-14` is fine (no brackets) but e.g. `h-[9px]`-style literal values exist elsewhere. Run `npx tsx scripts/audit-ui-drift.ts` (or a temporary broadened regex) to get the exact current list before deciding.

### Pitfall 2: Phase 31 already negotiated permanent exceptions — but only as prose comments, not `eslint-disable` directives
**What goes wrong:** `31-CONTEXT.md` D-05/D-06/D-08 already lists specific files as *intentionally* keeping raw elements or hardcoded hex — but the inline comments left at those sites (verified: `CategoryForm.tsx:34-35`, `CategoryTreeEditor.tsx:470-471`) are plain prose ("Do not map to a Tailwind CSS-variable token — see 31-CONTEXT.md D-08"), which does **nothing** to suppress an ESLint rule. Turning on the new rules without converting these to real suppression will break the build at these exact, already-approved sites.
**Why it happens:** Phase 31 wrote the comment as a forward-looking note for "whoever builds Phase 35," not as working code.
**How to avoid:** This phase's plan must include a task that converts each documented exception into an actual `// eslint-disable-next-line <rule-name> -- <reason, cite the decision id>` at the exact cited locations:
- `type="color"`/`"time"`/`"date"`/`"file"` raw inputs (D-05/D-07): `CategoryTreeEditor.tsx`, `CategoryForm.tsx`, `ComboAvailabilityEditor.tsx` (×2 time inputs), `AuditLogFilterBar.tsx` (×2 date inputs) — **recommend building this as a selector-level exemption instead of per-site disables**: `JSXOpeningElement[name.name='input']:not(:has(JSXAttribute[name.name='type'][value.value=/^(color|time|date|file)$/]))` so the rule never fires on these types anywhere, matching D-05/D-07's stated permanent, type-based exception.
- Signed-delta number input (D-06): `InventoryPagePanel.tsx:353` (`id="batch-delta"`) — no clean AST-selector exemption exists for "this specific number input"; needs a real `eslint-disable-next-line` comment at that exact line (or a shared `data-drift-exempt` attribute convention if the planner wants something more structural — Claude's discretion per CONTEXT.md).
- `category.color` hex values (D-08): `CategoryTreeEditor.tsx:472`, `CategoryForm.tsx:36,138` — need real `eslint-disable-next-line` comments; no clean selector-level exemption exists (these are ordinary string literals, indistinguishable by AST shape from a true drift hex).

### Pitfall 3: A live raw `<button>` exists in an undocumented, un-audited feature (`manage-promotions`)
**What goes wrong:** `src/features/manage-promotions/ui/PromotionAvailabilityEditor.tsx:91` has a raw `<button>` (day-of-week toggle chip) and two raw `type="time"` inputs (`:116,128`) — near-identical to the already-fixed `ComboAvailabilityEditor.tsx` pattern, but this file is **not** mentioned in Phase 31's `31-CONTEXT.md` D-04 list, and "promotions" doesn't appear in CLAUDE.md's Implemented Features list or `.planning/REQUIREMENTS.md` — it's a feature slice that landed outside this milestone's tracked scope, after (or alongside) Phase 33.
**Why it happens:** Phase 29's audit ran once, at a fixed point in time; any feature work landing after that point was never scanned.
**How to avoid:** D-14's pre-flip `npm run lint` checkpoint will catch this automatically **if** the rule scope includes `src/features/**` (which it must, per D-12) — the plan needs an explicit remediation task (swap the raw `<button>` for `POSButton`, matching `COMPONENT-01`'s existing pattern) before the checkpoint can pass, OR the checkpoint needs to surface this as a real, unplanned scope item rather than silently blocking. The two `type="time"` inputs are covered by the type-based exemption in Pitfall 2 and need no fix, only correct rule scoping.
**Warning signs:** `npx tsx scripts/audit-ui-drift.ts` currently reports 1 raw-button file / 7 raw-input files / 2 hardcoded-hex files remaining (down from Phase 29's original 20/8/3) — this is the authoritative live count as of 2026-07-15, **not** the stale numbers still committed in `.planning/phases/29-ui-drift-audit/DRIFT-AUDIT.md` (that file reflects Phase 29's original pre-fix state and was never regenerated after Phases 30-33 landed).

### Pitfall 4: `no-custom-classname` will flag `tw-animate-css` utility classes unless whitelisted
**What goes wrong:** The project depends on `tw-animate-css` (`@import 'tw-animate-css';` in `globals.css`) and uses classes like `animate-in`, `fade-in`, `slide-in-from-*` in ~12 files. `no-custom-classname` treats any class not recognized by core Tailwind + the project's `tailwind.config.ts` extensions as an error by default, with no automatic allowance for classes contributed by a separate imported CSS package.
**Why it happens:** The plugin statically resolves valid classnames from `tailwind.config.ts` + Tailwind core; it has no way to know about classes injected by a third `@import`ed stylesheet unless told.
**How to avoid:** Add a `whitelist` regex (e.g. `^(animate|fade|slide|zoom)-(in|out)(-from-\w+)?$`) to the plugin's `settings.tailwindcss` — confirm the exact class list used in this codebase (`grep -rn "animate-in\|fade-in\|slide-in-from\|zoom-in" src`) before finalizing the pattern, since `tw-animate-css`'s naming surface is broader than the 4 examples checked in this research pass.
**Warning signs:** Any `npm run lint` failure whose message is specifically about an unrecognized classname that visually matches an animation utility, not a real typo.

### Pitfall 5: `cn()` is not in the plugin's default `callees` list
**What goes wrong:** This project's `cn()` helper (`src/shared/lib/utils.ts` — `clsx` + `tailwind-merge` wrapper, the standard shadcn pattern) is used as `className={cn(...)}` in 11 files within `pages/widgets/features` [VERIFIED: `grep -rn "className={cn(" src/pages src/widgets src/features`]. `eslint-plugin-tailwindcss`'s default `callees` option is `['classnames', 'clsx', 'ctl', 'cva', 'tv']` [VERIFIED: source `docs/rules/no-arbitrary-value.md`, v3.18.3] — `cn` is absent, so string-literal arguments inside `cn(...)` calls are invisible to all three plugin rules unless `callees` is explicitly extended.
**Why it happens:** `cn` is a project-local convention name, not one the plugin can guess.
**How to avoid:** Set `settings.tailwindcss.callees: ['cn', 'clsx', 'classnames', 'ctl', 'cva', 'tv']` (see Pattern 2 example).
**Warning signs:** A known-bad arbitrary value or custom classname inside a `cn(...)` call in one of those 11 files does not get flagged even after the rules are enabled — silent gap, not a loud failure.

## Open Questions

1. **How should the arbitrary-spacing category (`no-arbitrary-value`) be scoped to avoid flagging the ~70 pre-existing non-touch-target `h-[...]`/`w-[...]`/`min-h-[...]`/`min-w-[...]` classes that were never in TOKEN-02's original (spacing-prefix-only) scope?**
   - What we know: D-08 explicitly locks in `no-arbitrary-value` as one of the two enforcement mechanisms ("Both... AND custom no-restricted-syntax rules"). The plugin rule has no per-property-category filter — it matches any bracketed value.
   - What's unclear: Whether the planner should (a) accept the larger footprint and add per-site `eslint-disable-next-line` comments to all ~70 legitimate non-drift arbitrary size values (large diff, but comments only — arguably compatible with "no visual changes"), or (b) implement the arbitrary-*spacing*-specifically check as a **custom** `no-restricted-syntax` regex mirroring Phase 29's exact prefix list (`p|px|py|...|space-y` only) and skip enabling `no-arbitrary-value` at all, treating D-08's "AND" as satisfied by `no-custom-classname` + `enforces-shorthand` (which are narrower and safer) plus the custom regex for spacing specifically.
   - Recommendation: Option (b) is lower-risk and stays truer to what was actually audited/fixed (TOKEN-02's literal scope), but is a deviation from the literal list of 3 sub-rules named in D-08. This is exactly the kind of tradeoff CONTEXT.md's "Claude's Discretion: exact regex patterns... implementer's judgment" was meant to cover — but given the scale (70 sites vs. a scope narrowing), the planner should surface this explicitly as a checkpoint/decision point rather than silently picking one, since it materially changes what "done" looks like for LINT-01.

2. **Does the planner want a structural, reusable exemption pattern (e.g., a `data-drift-exempt="reason"` attribute + selector) instead of scattered `eslint-disable-next-line` comments for the ~7 known permanent-exception sites?**
   - What we know: The 4 categories of known exceptions (native input types, signed-delta number, category-color hex, and — if option (a) above is chosen — ~70 non-drift size classes) all need some suppression mechanism.
   - What's unclear: Whether a single reusable pattern is worth the abstraction for a phase whose own framing is "documentation + enforcement only, no new tokens invented."
   - Recommendation: Plain `eslint-disable-next-line <rule> -- <decision-id>` comments are sufficient and match the existing codebase convention (`eslint-disable` is already used file-level in several places per CLAUDE.md's "Missing generated types workaround"). No new pattern needed.
