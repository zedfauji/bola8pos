# Phase 35: Guardrails — Tokens Doc & Drift Lint - Context

**Gathered:** 2026-07-15
**Status:** Ready for planning

<domain>
## Phase Boundary

Document the existing design tokens (colors/spacing/typography already in `tailwind.config.ts`/`index.css`, plus the touch-target and focus-ring conventions established in Phases 30-33) in a new reference doc, and add a drift-detection ESLint rule set so future code can't silently reintroduce the violations Phase 29 catalogued and Phases 30-33/33.1 fixed. Documentation + enforcement only — no new tokens invented, no visual/design changes.

</domain>

<decisions>
## Implementation Decisions

### Token doc scope & content
- **D-01:** `DESIGN-TOKENS.md` covers both tailwind.config.ts/index.css tokens (colors, spacing, typography, border-radius) AND the touch-target (44/56/72px) + `focusEmphasis` conventions from Phases 30-33, even though the latter are defined in component code (`POSButton.tsx`, `button.tsx`), not tailwind config — single reference doc for all UI consistency rules from this milestone.
- **D-02:** Each token category includes short do/don't usage examples (e.g. "use `bg-primary`, not `bg-blue-600`") — directly actionable, doubles as the rationale for the lint rule.
- **D-03:** One-line note on dark mode: colors are `var(--x)` CSS vars that already resolve correctly per-theme (dark mode is `media`, auto, no manual toggle) — no separate dark-mode token list needed.
- **D-04:** One-line cross-reference note: "These conventions are enforced by `eslint-rules/no-ui-drift.js`" — tells readers this isn't just documentation, it's actively checked.

### Token doc location & generation
- **D-05:** File location: `bar-pos/DESIGN-TOKENS.md` (repo root, matches the existing `FSD-STRUCTURE.md`/`DOMAIN-CONTRACTS.md`/`SUPABASE-CONTRACTS.md` pattern).
- **D-06:** The token tables (colors/spacing/typography/border-radius, i.e. the parts derivable from `tailwind.config.ts`/`index.css`) are generated by a script, `scripts/generate-design-tokens.ts` (matches the `scripts/audit-ui-drift.ts` convention from Phase 29), run manually via `npm run docs:tokens` — not auto-run in CI, not a git hook. Re-run when `tailwind.config.ts`/`index.css` changes.
- **D-07:** The generator only writes between `<!-- GENERATED:START -->` / `<!-- GENERATED:END -->` markers in `DESIGN-TOKENS.md`. The do/don't examples (D-02), dark-mode note (D-03), touch-target/focus-ring conventions (D-01's non-tailwind-config half), and the lint cross-reference (D-04) are hand-written outside the markers and untouched by regeneration.

### Lint mechanism & category coverage
- **D-08:** Both `eslint-plugin-tailwindcss` (new dependency — `no-arbitrary-value`/`no-custom-classname`/`enforces-shorthand` rules, covers arbitrary-spacing drift) AND custom `no-restricted-syntax` rules (covers raw `<button>`, raw `<input>`, hardcoded hex/rgb color literals — categories the plugin does not detect). Confirmed explicitly after a scope-mismatch check: the plugin alone cannot reach all 4 categories.
- **D-09:** All 4 of Phase 29's drift categories are enforced: raw `<button>`, raw `<input>`, hardcoded hex/rgb colors, arbitrary-value Tailwind spacing classes. The codebase already conforms on all 4 post Phases 30-33/33.1 — enforcing the full set now prevents regression on every category, not a subset.
- **D-10:** The 3 custom `no-restricted-syntax` rules live in a separate file, `eslint-rules/no-ui-drift.js`, imported into `eslint.config.js` (flat config) — keeps the UI-drift-specific rules grouped and `eslint.config.js` shorter.

### Severity & exemptions
- **D-11:** Severity is `error` (not warn-first) — blocks `npm run lint` / CI, matches CLAUDE.md's `max-warnings: 0` convention. The codebase already conforms (that's why this phase is sequenced last), so a warn-first soft-rollout period is unnecessary friction.
- **D-12:** Exemption: `shared/ui/*.tsx` only — the primitive-definition layer (`POSButton`, `Input`, etc. are DEFINED here using raw elements internally) is the sole legitimate exception. Everything else (`pages/`, `widgets/`, `features/`) must route through `shared/ui` primitives.

### CI integration & rollout safety
- **D-13:** New rules fold into the existing `eslint.config.js`, already run by `npm run lint` and already CI-gated — no new CI step/pipeline change needed.
- **D-14:** Before flipping severity to `error`, this phase runs `npm run lint` as a verification task/checkpoint against the full codebase to confirm zero violations exist right now — a cheap safety check against drift that may have crept back in since Phase 33's sweep (e.g. from Phase 34's new visual-test files or any hotfixes).

### Post-research amendments (resolved 2026-07-15, after RESEARCH.md findings)
- **D-15:** Arbitrary-spacing drift (TOKEN-02) is enforced via a **narrow custom `no-restricted-syntax` regex** matching Phase 29's original `p-[...]`/`m-[...]`/`gap-[...]` prefix scope — NOT `eslint-plugin-tailwindcss`'s `no-arbitrary-value` rule, which flags any bracketed value (incl. `h-[...]`/`w-[...]`/`min-h-[...]`) and would sweep in ~70 pre-existing, never-audited arbitrary values across ~40 files outside this phase's scope. `eslint-plugin-tailwindcss@3.18.3` is still added for its other rules (`no-custom-classname`, `enforces-shorthand`) per D-08, just not `no-arbitrary-value`.
- **D-16:** The undocumented raw `<button>` (day-picker) in `src/features/manage-promotions/ui/PromotionAvailabilityEditor.tsx` — found by RESEARCH.md, landed after Phase 29's audit, never fixed — is fixed in this phase (swap for `POSButton`/`Button`, matching Phases 30-33's pattern) so D-14's "npm run lint passes clean" checkpoint is genuinely true, not pre-suppressed.

### Claude's Discretion
- Exact wording/table formatting inside `DESIGN-TOKENS.md`'s generated-block markers.
- Exact regex patterns for the 3 custom `no-restricted-syntax` rules — implementer's judgment, informed by `scripts/audit-ui-drift.ts`'s existing patterns from Phase 29.
- Ordering of tasks within the phase's wave(s).
