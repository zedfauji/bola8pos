## Spacing Scale

Source of truth: `tailwind.config.ts` does **not** override Tailwind's default `spacing` scale (no `theme.extend.spacing` block exists) ŌĆö the project uses Tailwind's built-in 4px-increment scale unmodified. `scripts/generate-design-tokens.ts` should document the subset actually observed in use, e.g.:

| Token | Value | Usage (observed in codebase) |
|-------|-------|-------------------------------|
| `1` | 4px | Icon gaps, tight inline padding |
| `2` | 8px | Compact element spacing |
| `4` | 16px | Default element spacing (default `p-4`/`gap-4`) |
| `6` | 24px | Section padding |
| `8` | 32px | Layout gaps |
| `12` | 48px | Major section breaks |
| `16` | 64px | Page-level spacing |

Exceptions (pre-existing, documented not invented):
- `POSButton`/`Button` touch-target floors are expressed as `min-h-[44px]` / `min-h-[56px]` / `min-h-[72px]` ŌĆö these are **functional touch-target minimums**, not spacing-scale tokens. `TOKEN-02`'s arbitrary-value ban (enforced by `eslint-plugin-tailwindcss`'s `no-arbitrary-value` rule) must **not** flag these three `min-h-[Npx]` values ŌĆö they are the canonical touch-target contract from Phases 30-33, not drift. `eslint-rules/no-ui-drift.js` / the plugin config needs an explicit allowlist/ignore for `min-h-[44px]`, `min-h-[56px]`, `min-h-[72px]` (and any exact same-value siblings) inside `shared/ui/POSButton.tsx` and wherever those exact literals are reused. Verify this exception is wired before flipping severity to `error` (D-14's lint-clean checkpoint will otherwise self-fail).

## Color

Source of truth: `src/app/globals.css` (`:root` + `@media (prefers-color-scheme: dark)` blocks) and `tailwind.config.ts`'s `theme.extend.colors` map. All values are `oklch()` CSS custom properties, not hex ŌĆö consumed via Tailwind utility classes (`bg-primary`, `text-foreground`, etc.), never `hsl()`-wrapped.

| Role | Token(s) | Usage |
|------|----------|-------|
| Dominant | `--background` / `--foreground` | Page background, default text |
| Secondary | `--card`, `--secondary`, `--muted`, `--popover` | Cards, panels, secondary surfaces |
| Accent (interactive) | `--primary` | Primary CTA buttons only (existing usage; not a new reservation) |
| Destructive | `--destructive` | Destructive actions only (void, delete, cancel) |
| Semantic (POS-specific) | `--pos-accent` (green), `--pos-danger` (red) | Table-status semantics only ŌĆö "available" / "occupied/danger" ŌĆö not general-purpose accent colors |

Dark mode: `darkMode: 'media'` ŌĆö follows OS `prefers-color-scheme`, no manual toggle. All tokens above resolve automatically per-theme via the CSS-variable indirection (matches CONTEXT.md D-03 ŌĆö no separate dark-mode token list needed in `DESIGN-TOKENS.md`).

`TOKEN-01` (no hardcoded hex/rgb) is what `eslint-rules/no-ui-drift.js` enforces here ŌĆö the contract is "use one of the tokens above via its Tailwind class," not a new palette decision.

## Copywriting Contract

No new user-facing UI copy this phase (no dialogs, no empty states, no CTAs). The one real copywriting surface is the **lint rule error messages** developers will see when `eslint-rules/no-ui-drift.js` / `eslint-plugin-tailwindcss` fires. These should be specific enough to self-serve a fix without opening `DESIGN-TOKENS.md`:

| Rule | Message |
|------|---------|
| Raw `<button>` outside `shared/ui/` | `Use POSButton or Button from @shared/ui instead of a raw <button> element. See DESIGN-TOKENS.md.` |
| Raw `<input>` outside `shared/ui/` | `Use FormField, MoneyInput, or the correct shared/ui input primitive instead of a raw <input> element. See DESIGN-TOKENS.md.` |
| Hardcoded hex/rgb color literal | `Use a Tailwind CSS-variable token (e.g. bg-primary, text-foreground) instead of a hardcoded color value. See DESIGN-TOKENS.md.` |
| Arbitrary-value spacing (`eslint-plugin-tailwindcss` `no-arbitrary-value`) | Plugin default message; do not override unless it proves unclear in practice ŌĆö exception noted above for the 3 touch-target `min-h-[Npx]` literals. |

`shared/ui/*.tsx` is exempt from all of the above (D-12) ŌĆö these rules only fire in `pages/`, `widgets/`, `features/`.

No empty-state, error-state, or destructive-confirmation copy applies ŌĆö this phase adds no dialogs or data views.
