**Step B — Create .github/workflows/release.yml:**

Create the directory `.github/workflows/` if it does not exist, then create `release.yml` with this exact content (replace `YOUR_ORG/YOUR_REPO` with the actual GitHub org/repo from `git remote get-url origin`):

```yaml
name: publish
on:
  workflow_dispatch:
  push:
    tags:
      - 'v*'

jobs:
  publish-tauri:
    permissions:
      contents: write
    runs-on: windows-latest
    defaults:
      run:
        working-directory: bar-pos

    steps:
      - uses: actions/checkout@v4

      - name: setup node
        uses: actions/setup-node@v4
        with:
          node-version: lts/*
          cache: npm
          cache-dependency-path: bar-pos/package-lock.json

      - name: install Rust stable
        uses: dtolnay/rust-toolchain@stable

      - name: Rust cache
        uses: swatinem/rust-cache@v2
        with:
          workspaces: 'bar-pos/src-tauri -> target'

      - name: install frontend dependencies
        run: npm ci

      - uses: tauri-apps/tauri-action@v0
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          TAURI_SIGNING_PRIVATE_KEY: ${{ secrets.TAURI_SIGNING_PRIVATE_KEY }}
          TAURI_SIGNING_PRIVATE_KEY_PASSWORD: ${{ secrets.TAURI_SIGNING_PRIVATE_KEY_PASSWORD }}
        with:
          tagName: v__VERSION__
          releaseName: 'Bar POS v__VERSION__'
          releaseBody: 'See CHANGELOG.md for release notes.'
          releaseDraft: true
          prerelease: false
          includeUpdaterJson: true
          projectPath: .
```

Note: `working-directory: bar-pos` is set in `defaults.run` — the `projectPath: .` in `tauri-action` means it looks for `tauri.conf.json` at `bar-pos/src-tauri/tauri.conf.json` relative to the working-directory.

**Step C — Bump version in tauri.conf.json for updater testing:**

The Tauri updater compares the running app's version (from `tauri.conf.json`) against the version in `latest.json` from GitHub Releases. If they are equal, `check()` returns `null` (no update detected). To verify the updater dialog fires during development, bump the local version to `"1.1.0"` before publishing a test release:

Find `"version"` in `bar-pos/src-tauri/tauri.conf.json` (currently `"1.0.0"`) and change it to `"1.1.0"`:

```json
"version": "1.1.0"
```

Then publish a GitHub release tagged `v1.0.0` (the OLD version). When the app runs at `1.1.0` and the published release at `1.0.0` is visible via `latest.json`, the updater detects a newer local version and returns null — which is expected; the release tag must be HIGHER than the running version to trigger the dialog. So the correct test sequence is: bump app to `"1.1.0"`, publish release `v1.1.0` via git tag, then run the old `v1.0.0` build pointing at that endpoint. For a simpler local smoke-test: set local version to `"0.0.1"`, publish a release at `v1.0.0` — check() will return the update object.

**Convention going forward:** Always increment `tauri.conf.json` version before tagging a release. The release tag (`v*`) must match the version value in `tauri.conf.json`.
