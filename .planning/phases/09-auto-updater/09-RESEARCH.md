## Summary

Phase 9 integrates Tauri 2's first-party `tauri-plugin-updater` to deliver in-app update detection against GitHub Releases. The update lifecycle is: Rust plugin checks the `latest.json` endpoint (generated by `tauri-action` with `includeUpdaterJson: true`), returns an update object to the JS layer via `@tauri-apps/plugin-updater`, the React hook exposes state to `UpdateAvailableDialog`, and on user confirmation `downloadAndInstall()` streams progress events before `relaunch()` restarts the app.

The entire signing chain is: `npm run tauri signer generate` produces an Ed25519 key pair → private key stored as `TAURI_SIGNING_PRIVATE_KEY` GitHub Actions secret → public key embedded in `tauri.conf.json` `plugins.updater.pubkey` → `tauri-action` uses the private key at build time to sign `.exe`/`.msi` artifacts and generate `.sig` files → updater verifies `.sig` at install time.

The four-state dialog (`idle → downloading → restart-ready → error`) is fully specced in `09-UI-SPEC.md`. A `Progress` shadcn component needs to be added via CLI before dialog implementation. All other shadcn components already exist.

**Primary recommendation:** Follow the serial plan order — signing first (09-01), plugin wiring second (09-02), React hook third (09-03), dialog fourth (09-04), wiring + tests fifth (09-05). Each step is a hard dependency on the previous.

### Pattern 4: JS `check()` with progress and relaunch
**What:** Full update check → download → install → restart sequence.
**When to use:** Inside `useAppUpdater.ts`.
```typescript
// Source: https://v2.tauri.app/plugin/updater/
import { check } from '@tauri-apps/plugin-updater';
import { relaunch } from '@tauri-apps/plugin-process';

const update = await check();
if (update) {
  let downloaded = 0;
  let contentLength = 0;
  await update.downloadAndInstall((event) => {
    switch (event.event) {
      case 'Started':
        contentLength = event.data.contentLength ?? 0;
        break;
      case 'Progress':
        downloaded += event.data.chunkLength;
        const percent = contentLength > 0
          ? Math.round((downloaded / contentLength) * 100)
          : 0;
        setPercent(percent);
        break;
      case 'Finished':
        setPercent(100);
        break;
    }
  });
  await relaunch();
}
```

### Pattern 5: useAppUpdater Hook Structure
**What:** Startup check + 4-hour interval with cleanup. Follows `useServerTimeDrift` pattern.
**When to use:** Plan 09-03.
```typescript
// Source: codebase pattern (useServerTimeDrift.ts, usePersistedBool.ts)
import { useEffect, useState, useRef } from 'react';
import { check } from '@tauri-apps/plugin-updater';
import { relaunch } from '@tauri-apps/plugin-process';
import { logger } from '@shared/lib/logger-instance';

const FOUR_HOURS_MS = 4 * 60 * 60 * 1000;

export type UpdaterState =
  | { phase: 'idle' }
  | { phase: 'available'; version: string; changelog: string }
  | { phase: 'downloading'; version: string; percent: number }
  | { phase: 'restart-ready'; version: string }
  | { phase: 'error' };

export function useAppUpdater() {
  const [state, setState] = useState<UpdaterState>({ phase: 'idle' });
  const updateRef = useRef<Awaited<ReturnType<typeof check>>>(null);

  const runCheck = async () => {
    try {
      const update = await check();
      if (update) {
        updateRef.current = update;
        setState({
          phase: 'available',
          version: update.version,
          changelog: update.body ?? '',
        });
      }
    } catch (err) {
      logger.warn('updater.check_failed', { err });
      // UPD-07: silent failure — no UI shown
    }
  };

  useEffect(() => {
    void runCheck();
    const interval = setInterval(() => { void runCheck(); }, FOUR_HOURS_MS);
    return () => clearInterval(interval);
  }, []);

  const startInstall = async () => {
    if (!updateRef.current || state.phase !== 'available') return;
    const { version } = state;
    setState({ phase: 'downloading', version, percent: 0 });
    try {
      await updateRef.current.downloadAndInstall((event) => {
        if (event.event === 'Progress') {
          // percent computed from event.data
        }
        if (event.event === 'Finished') {
          setState({ phase: 'restart-ready', version });
        }
      });
    } catch (err) {
      logger.error('updater.install_failed', { err });
      setState({ phase: 'error' });
    }
  };

  const dismissUpdate = () => setState({ phase: 'idle' });

  return { state, startInstall, dismissUpdate, relaunch };
}
```

### Pattern 8: Key Generation Command
**What:** Generate Ed25519 signing key pair.
**When to use:** One-time operation in Plan 09-01.
```bash
# Source: https://v2.tauri.app/plugin/updater/
# Run from bar-pos/ directory
npm run tauri signer generate -- -w ~/.tauri/bola8pos.key

#   Private key: ~/.tauri/bola8pos.key         → copy content to GH secret TAURI_SIGNING_PRIVATE_KEY
```

**Critical:** Never commit the private key. Store it ONLY in GitHub Actions secret. If the private key is lost, existing installs cannot receive updates.

### Anti-Patterns to Avoid
- **Silent install without user confirm:** CONTEXT.md forbids it — always show dialog first.
- **`setInterval` without cleanup:** Always return `() => clearInterval(interval)` from `useEffect`.
- **Calling `relaunch()` during active service:** Dialog state machine must reach `restart-ready` before calling relaunch; "Later" button lets bartenders defer.
- **Logging the private key:** Never log `TAURI_SIGNING_PRIVATE_KEY` value — structured logger must never receive it as a field.
- **`installMode: "quiet"` on Windows:** Requires admin privilege the app may not have; use `"passive"` instead.
- **Committing `.key` file:** gitignore `*.key` and `*.key.pub` files proactively.
- **Triggering check in a component (not provider):** Check runs in `UpdaterProvider` mounted at root — not in individual pages/widgets that may unmount.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Semver comparison | Custom version parser | tauri-plugin-updater (Rust does it) | Edge cases: pre-release tags, build metadata |
| Artifact download with retry | Custom fetch loop | `update.downloadAndInstall()` | Plugin handles TLS, redirect, partial download, signature verification |
| Ed25519 signature verification | Custom crypto | Plugin verifies `.sig` automatically against `pubkey` | Cryptographic implementation errors are catastrophic for update security |
| OS installer execution | `std::process::Command` calling msiexec | Plugin's `download_and_install` | Handles UAC elevation, NSIS/MSI detection, passive mode on Windows |
| App restart | `std::process::exit()` | `relaunch()` from `tauri-plugin-process` | Proper Tauri app shutdown sequence; avoids orphaned processes |

**Key insight:** The entire download-verify-install-restart pipeline must be owned by Tauri's official plugin. Any custom code in this path creates a security vulnerability (unsigned updates) or a Windows compatibility failure (UAC, installer type detection).

## Common Pitfalls

### Pitfall 1: `check()` Throws on No Update vs. Returns Null
**What goes wrong:** Developers expect `check()` to throw when there's no update; it actually returns `null`. Wrapping in try/catch but only checking the error branch misses the "no update" case.
**Why it happens:** Confusion with the v1 updater API which had different semantics.
**How to avoid:** Always check: `const update = await check(); if (update) { ... }`. The try/catch is only for genuine network failures.
**Warning signs:** Dialog never shows even after bumping version in `tauri.conf.json` during testing.

[VERIFIED: https://v2.tauri.app/plugin/updater/]

### Pitfall 2: Private Key Lost = No More Updates
**What goes wrong:** Key pair generated locally, private key not securely backed up, dev machine wiped. Existing installs cannot receive signed updates — all users must manually reinstall.
**Why it happens:** One-time setup step done quickly without backup.
**How to avoid:** Immediately after `signer generate`: (1) add to GH secret, (2) store private key in a password manager or encrypted vault. Document backup location in team wiki.
**Warning signs:** Build fails with "TAURI_SIGNING_PRIVATE_KEY not set" in CI, or updater plugin refuses to accept update artifacts.

### Pitfall 3: `createUpdaterArtifacts: true` Missing in tauri.conf.json
**What goes wrong:** `tauri-action` runs successfully but no `.sig` files or `latest.json` are generated/uploaded to the release. The endpoint returns 404 and every `check()` call silently fails.
**Why it happens:** The bundle config change is easy to forget when the plugin registration is done in a separate step.
**How to avoid:** Plan 09-02 explicitly adds `bundle.createUpdaterArtifacts: true` to `tauri.conf.json` as its first action.
**Warning signs:** GitHub release assets list shows no `latest.json` file.

### Pitfall 4: `includeUpdaterJson: true` Missing in tauri-action
**What goes wrong:** Even with `createUpdaterArtifacts: true`, the `latest.json` may not be uploaded to the release if `includeUpdaterJson: true` is omitted from the tauri-action `with:` block.
**Why it happens:** The option is a tauri-action concern, not a Tauri build concern — easy to miss.
**How to avoid:** Always set `includeUpdaterJson: true` in the release workflow.
**Warning signs:** `.sig` files appear in release assets but `latest.json` is absent.

[CITED: https://ratulmaharaj.com/posts/tauri-automatic-updates/]

### Pitfall 5: Windows `installMode` Default is Not `passive`
**What goes wrong:** Without explicit `installMode` config, Windows updater may attempt to run the installer in interactive mode, presenting a full NSIS wizard to the bartender mid-shift.
**Why it happens:** Tauri docs state `passive` is "recommended default" but it is NOT the built-in default.
**How to avoid:** Always set `"plugins": { "updater": { "windows": { "installMode": "passive" } } }` in `tauri.conf.json`.
**Warning signs:** Clicking "Install Now" launches a Windows installer window outside the app.

[VERIFIED: https://v2.tauri.app/plugin/updater/]

### Pitfall 6: `setInterval` Drift Over Long Sessions
**What goes wrong:** `setInterval` is not guaranteed to fire exactly every 4 hours in a Tauri WebView2 context if the system suspends/hibernates. The interval may fire very late after resume.
**Why it happens:** Windows power management can throttle background timers.
**How to avoid:** For a POS running 12-18 hours/day, this is an acceptable known limitation (UPD-02 says "approximately" 4 hours). If precision is critical, use Tauri's system event listeners. For this phase: `setInterval` is sufficient.
**Warning signs:** None in normal operation — only noticed after sleep/hibernate which is unusual for a POS terminal.

### Pitfall 7: `updater:default` Permission Not Added to Capabilities
**What goes wrong:** `check()` throws a Tauri permission error ("Command updater_check is not allowed") because the capability was not granted.
**Why it happens:** Tauri 2's capability system blocks all plugin commands by default — opt-in per command.
**How to avoid:** Plan 09-02 adds `"updater:default"` and `"process:allow-relaunch"` to `capabilities/default.json`.
**Warning signs:** Console shows a Tauri IPC error mentioning "not allowed" when the app loads.

[VERIFIED: https://v2.tauri.app/plugin/updater/]

### Pitfall 8: Dialog Dismissible During Download
**What goes wrong:** User closes the dialog mid-download; download continues in background; `relaunch()` is called but dialog is gone — user sees the app restart without warning.
**Why it happens:** Default Radix `Dialog` allows Escape to dismiss.
**How to avoid:** Dialog contract in `09-UI-SPEC.md` specifies `onOpenChange` is a no-op during the `downloading` state. Implement by checking state in the `onOpenChange` handler.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Tauri v1 built-in updater (in core) | `tauri-plugin-updater` (separate plugin) | Tauri v2 migration | Must add plugin explicitly; v1 config shape different |
| `tauri.conf.json` `updater` at root | `tauri.conf.json` `plugins.updater` nested | Tauri v2 | Config path changed; v1 examples are misleading |
| `update.install()` + manual restart | `update.downloadAndInstall()` + `relaunch()` from process plugin | Tauri v2 | API consolidated; restart now requires separate process plugin |

**Deprecated/outdated:**
- Tauri v1 updater config shape (`tauri.conf.json` root-level `updater` key): replaced by `plugins.updater` in v2
- `update.install()` standalone (v1): use `update.downloadAndInstall(progressCb)` in v2
- `emit`/`listen` for update events (v1 pattern): v2 uses callback-based `downloadAndInstall`

## Security Domain

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | no | — |
| V3 Session Management | no | — |
| V4 Access Control | no | — |
| V5 Input Validation | yes | `update.version` displayed in UI — render as text, never innerHTML |
| V6 Cryptography | yes | Ed25519 signing via `tauri signer generate`; verification is handled by tauri-plugin-updater |
| V10 Malicious Code / Supply Chain | yes | Only official tauri-plugin-updater; no community crates; TAURI_SIGNING_PRIVATE_KEY stored only in GH Secrets |

### Known Threat Patterns for Tauri Updater Stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Unsigned update artifact | Tampering | Ed25519 signature required; plugin rejects unsigned artifacts |
| Private key leakage | Elevation of Privilege | Store ONLY in GitHub Secrets; never log; gitignore `.key` files |
| Downgrade attack | Tampering | Tauri updater only installs versions > current version (semver enforced by plugin) |
| Man-in-the-middle on `latest.json` fetch | Tampering | HTTPS endpoint + `.sig` signature verification — plugin validates signature against embedded pubkey |
| Malicious changelog markdown (XSS) | Tampering | Render `update.body` as `whitespace-pre-wrap` text (per UI-SPEC), not `dangerouslySetInnerHTML` |
